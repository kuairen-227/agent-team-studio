# 0007. AI駆動開発アーキテクチャ

## Status

accepted

- 作成日: 2026-04-17
- 関連: ADR-0002（前提）, Issue #26

## Context

本プロジェクトでは 1 人のエンジニアが Claude Code と協働し、企画→要件定義→設計→開発→試験→改善→運用保守の全工程をカバーする（ADR-0002）。

現在、Claude Code のスキルとして 4 つが整備されている（create-adr, create-issue, create-pr, process-issue）。これらは特定タスクの自動化であり、開発ライフサイクル全体で必要な専門的役割をカバーしていない。

方針が未定のまま開発フェーズに入ると、場当たり的にスキルや運用が増え、一貫性を失うリスクがある。エージェントアーキテクチャ、人間の関与モデル、品質保証の仕組み、Claude Code の活用範囲の 4 観点で方針を定める。

## Considered Alternatives

### 観点 1: エージェントアーキテクチャ

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | スキル拡張方式 — 役割別にスキルを追加（review-code, design-api 等） | 却下 — スキル数が膨張し管理コストが増大する |
| B | ロールスイッチ方式 — プロンプトで役割を切り替える | 却下 — 暗黙知に依存し再現性が低い |
| C | マルチエージェント方式 — sub-agent で異なる役割を並行稼働 | 却下 — 1 人チームでは制御コストが便益を上回る |
| D | ハイブリッド方式 — スキル＋ロール定義の組合せ、必要に応じて sub-agent | **採用** |

### 観点 2: 人間の関与モデル

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | 全承認モデル — すべての AI 出力を人間がレビュー | 却下 — オーバーヘッドが大きく開発速度を著しく低下させる |
| B | チェックポイントモデル — 重要な判断ポイントのみ承認、日常作業は自律実行 | **採用** |
| C | 例外ベースモデル — AI が自律実行し、迷った場合のみエスカレーション | 却下 — 学習目的の意思決定機会が減少する |

### 観点 3: 品質保証の仕組み

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | プロセスベース — テンプレート・チェックリスト・規約で品質担保 | 単独では却下 — 形式的遵守に留まりやすい |
| B | レビューベース — AI 生成物を別の AI ロールがレビュー | 却下 — 1 人チームで AI-on-AI レビューは検証コストに対して効果が不確実 |
| C | テスト駆動ベース — 自動テスト・CI で検証、不合格時に自動修正ループ | 単独では却下 — テストで捕捉できない設計・方針レベルの品質がある |
| D | 三層構成（プロセス + テスト駆動 + フィードバックループ） | **採用** |

### 観点 4: Claude Code の活用範囲

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | Skills 中心 — 役割ごとのスキルを充実させ、Hooks で自動化を補完 | **採用（初期）** |
| B | MCP Server 連携 — 外部ツールと MCP で連携し Claude Code をハブにする | 保留 — 外部ツール連携の必要性が生じてから導入 |
| C | CLI + CI 統合 — Claude Code CLI をローカル中心に、CI/CD にも AI チェック組込み | **採用（段階導入）** |

## Decision

### 1. エージェントアーキテクチャ: ハイブリッド方式

定型的な作業はスキルで効率化し、専門的な判断が必要な場面ではロール定義で視点を切り替える。

**スキル層（定型作業の自動化）**:

既存の 4 スキル（create-adr, create-issue, create-pr, process-issue）を基盤とし、開発フェーズの進行に応じて追加する。スキルの追加基準は「繰り返し実行され、手順が定型化できる作業」とする。

**ロール定義層（専門的視点の提供）**:

CLAUDE.md またはプロンプトでロールを切り替え、特定の専門性を持つ視点で作業する。ロールは固定的なペルソナではなく、作業コンテキストに応じて柔軟に適用する。

想定ロール例:

| ロール | 適用場面 | 主な活動 |
| --- | --- | --- |
| アーキテクト | 設計判断 | 技術選定、構造設計、ADR 作成 |
| 実装者 | コーディング | 実装、リファクタリング |
| QA | 品質検証 | テスト戦略、テストコード作成、品質分析 |
| レビュアー | コードレビュー | PR レビュー、改善提案 |
| PM 補佐 | 企画・管理 | 要件整理、優先順位付け、進捗整理 |

**sub-agent の利用方針**:

現時点では積極的に導入しない。以下の条件を満たす場合に個別判断で導入する:

- 独立した作業を並行実行する明確な利点がある
- 作業間の依存関係が少なく、統合コストが低い

### 2. 人間の関与モデル: チェックポイント方式

人間が判断するチェックポイントと、AI が自律実行する範囲を明確に分ける。

**人間が承認するチェックポイント**:

- 設計判断（ADR の内容）
- スコープに影響する要件変更
- 外部サービス・ライブラリの選定
- 作業計画（process-issue の計画提示）

**AI が自律実行する範囲**:

- 計画承認後の実装作業
- テストコードの作成・実行
- lint / format の自動修正
- コミット・ブランチ作成（規約に従う範囲）

**エスカレーション基準**:

AI は以下の場合にユーザーに判断を仰ぐ:

- 要件の解釈に曖昧さがある
- 複数の実装方針があり、トレードオフが明確
- 既存の ADR や規約と矛盾する状況

### 3. 品質保証: プロセス + テスト駆動 + フィードバックループ

3 つの仕組みを組み合わせて品質を担保する。

**第 1 層: プロセスベース（一貫性の確保）**

- テンプレート: Issue / PR / ADR テンプレートによる構造化
- 規約: CLAUDE.md のコーディング規約・コミット規約
- チェックリスト: PR テンプレートの確認項目

**第 2 層: テスト駆動（自動検証）**

- 単体テスト・統合テストによる機能検証
- CI での自動実行（lint, type-check, test）
- テストカバレッジの計測

**第 3 層: フィードバックループ（継続改善）**

検証結果を AI にフィードバックし、自動修正サイクルを回す仕組み。段階的に導入する:

| 段階 | 内容 | 時期 |
| --- | --- | --- |
| Phase 1 | Hooks で lint / type-check の結果を Claude Code に自動フィードバックし、修正ループを実行 | 初期から |
| Phase 2 | CI 失敗時の解析・修正提案を自動化 | CI 整備後 |
| Phase 3 | 実行メトリクス・ユーザーフィードバックの蓄積によるプロンプト・テンプレートの継続改善 | 運用安定後 |

### 4. Claude Code の活用方針: Skills 中心 + 段階的 CI 統合

**初期（現在〜MVP 開発中）**:

- Skills: 定型作業の自動化（既存 4 スキル + 開発進行に応じた追加）
- Hooks: フィードバックループ Phase 1 の実現（lint / type-check 結果の自動フィードバック）
- CLAUDE.md: ロール定義・規約の集約

**MVP 開発中盤以降**:

- CI 統合: GitHub Actions で自動テスト・品質チェックを実行し、結果を開発フローにフィードバック
- MCP Server: 外部ツール連携の必要性が生じた場合に個別導入

### 5. フェーズ別の役割と実現手段

| 工程 | 主な活動 | 実現手段 |
| --- | --- | --- |
| 企画 | 市場調査、競合分析、ロードマップ | ロール: PM 補佐 |
| 要件定義 | ユーザーストーリー、受け入れ条件 | ロール: PM 補佐 + Issue テンプレート |
| 設計 | アーキテクチャ、API、DB、UI/UX | ロール: アーキテクト + スキル: create-adr |
| 開発 | 実装、コードレビュー | ロール: 実装者 / レビュアー + スキル: process-issue |
| 試験 | テスト戦略、テストコード、品質分析 | ロール: QA + CI 自動検証 |
| 改善 | パフォーマンス分析、UX 改善 | ロール: QA / アーキテクト |
| 運用保守 | 監視、インシデント対応、ドキュメント保守 | CI 統合 + Hooks |
| 横断 | プロジェクト管理、リスク管理 | ロール: PM 補佐 + スキル: create-issue / create-pr |

### 6. 段階的導入計画

| フェーズ | 対応内容 |
| --- | --- |
| 現在 | 既存 4 スキル + CLAUDE.md 規約で運用 |
| 技術スタック選定〜初期実装 | ロール定義の整備、Hooks によるフィードバックループ Phase 1 導入 |
| MVP 開発中盤 | CI/CD 整備、テスト駆動の本格運用、必要に応じてスキル追加 |
| MVP 開発後半 | フィードバックループ Phase 2（CI 連携）、レビュアーロールの活用 |
| MVP リリース後 | フィードバックループ Phase 3（メトリクス蓄積）、運用保守ロールの活用 |

## Consequences

- スキル追加・ロール定義・品質施策に一貫した方針が生まれ、場当たり的な拡張を防げる
- チェックポイント方式により、学習目的の意思決定機会を確保しつつ開発速度を維持できる
- フィードバックループの段階導入により、初期は低コストで始め、プロジェクトの成熟に応じて品質保証を強化できる
- ハイブリッド方式はスキルとロールの境界判断が必要になるが、1 人チームでは柔軟に運用できる
- sub-agent を初期導入しないため、並行作業の効率化は限定的になる（必要時に個別導入で対応）
- #10（技術スタック選定）、#11（アーキテクチャ方針）、#12（CI/CD 整備）の進め方に本 ADR の方針が適用される
