# 0004. ターゲットユーザー

## Status

proposed

## Context

ADR-0003 でプロダクトコンセプトを定めた際、ターゲットユーザーは「AI に関心のある業務人材（企画・PdM・コンサル職等）」と仮置きされた。本 ADR では、この仮説を具体的なペルソナ／ユースケース／拡張ロードマップに落とし込み、#9 MVP スコープ判断の拠り所を提供する。

ペルソナ選定にあたっては、開発プロセス上の都合（ドッグフーディングのしやすさ）を選定の主軸に据えると作り手バイアスに引き戻されるリスクがあるため、**痛点起点の判断軸を明文化**する必要があった。

## Decision

### ペルソナ選定基準（3 軸）

ペルソナ候補は以下の 3 軸で評価する。**ドッグフーディング可能性は補助軸であり、プライマリ選定には用いない。**

1. **痛点の強度** — 課題が定常的・反復的に発生し、業務時間やアウトカムに直接影響するか
2. **課題の切実さ** — 「AI で解決したい」という能動的動機を持ちやすいか（= "AI 好き寄り" レンジに収まるか）
3. **MVP での検証可能性** — 短期の実装スコープで効果を定量的に観測できるか（例: 作業時間の削減幅）

### プライマリペルソナ: 社内新規事業企画担当（Beachhead）

- **所属**: 中堅〜大手事業会社の新規事業開発部 / DX 推進室
- **役割**: 市場調査、競合分析、仮説検証サイクルの実行
- **AI 習熟度**: ChatGPT は日常利用。API や自動化は未踏（"AI 好き寄り" レンジ）
- **JTBD（Job to be Done）**:
  > **When** 月次の事業戦略レビュー前、**I want to** 競合 3 社の直近動向を観点別（戦略 / 製品 / 投資 / パートナーシップ）に比較整理したい、**so I can** 役員提案の裏付けを **30 分以内**に固められる。現状は手動で 3 時間かけている。

### セカンダリペルソナ: エンジニア出身 PdM（ドッグフーディング兼用）

- **所属**: 自社プロダクトを持つ事業会社の PdM 組織
- **役割**: 要件定義・競合調査・ステークホルダー調整の下調べ
- **AI 習熟度**: 高い。API・自動化にも踏み込める
- **位置付け**: ビジネス価値提供の対象であると同時に、本プロジェクトチーム（PM + エンジニア）自身の近傍として設定することで、**開発中のドッグフーディングを成立させる補助役**

### 主要ユースケース（コア 3 本）

エージェントチーム化の必然性（= 単発 LLM では本質的に解けない / 大きく劣る）があるものに絞り込む。FAQ 整理・ADR 草稿生成は候補から外す（理由は後述）。

| # | ユースケース | 主対象 | チーム化の必然性 |
| - | --- | --- | --- |
| 1 | 競合調査の並列深掘り（観点別に複数競合を横比較） | プライマリ | 観点 × 競合の 2 軸マトリクスを**観点別調査役の並列起動**で埋める |
| 2 | 顧客・ステークホルダーフィードバック分析（定性データ整理） | プライマリ | **分類 / 感情 / 優先度**エージェントの分業で精度と網羅性が単発 LLM を大きく超える |
| 3 | 戦略レビュー前の論点整理・下調べ（アジェンダ草案生成） | プライマリ / セカンダリ | **議題抽出役 / 背景調査役 / 論点整理役**の分業により、単発 LLM の単層回答より構造が揃う |

### スコープ外（本 ADR で意図的に除外するユーザー像）

- **非エンジニア一般**（経理・総務など AI 活用動機が薄い層） — 痛点強度と切実さが基準を満たさない
- **エンタープライズ管理者**（IT 統制・監査・SSO 担当） — ADR-0003 のスコープ外と整合
- **開発者**（エンジニア本人のコーディング支援用途） — 北極星「業務人材の AI 活用民主化」から外れる

### ビーチヘッド戦略と拡張ロードマップ

プライマリを「新規事業企画担当」に絞るのは、北極星「誰もが、自分の AI エージェントチームを率いて働く」へ到達するための**ビーチヘッド**としての意図的選択である。

**選定理由**:

- 3 軸評価（痛点強度 / 切実さ / 検証可能性）がいずれも高水準で揃う唯一の候補
- 調査系タスクは作業時間で定量化しやすく、MVP 成果判定がシンプル
- ADR-0002「社内新規事業」シナリオと文脈が自然に接続する

**拡張ロードマップ**（学習プロジェクトのため時間軸は相対表現）:

| フェーズ | 対象層 | 次フェーズへの移行条件 |
| --- | --- | --- |
| ビーチヘッド | 新規事業企画担当 | MVP リリース＋プライマリ JTBD の完走確認 |
| 拡張 1 | PdM・マーケ企画（調査系タスクを持つ周辺職種） | コア UC で作業時間 **50% 以上削減**が定量的に観測される |
| 拡張 2 | コンサル・リサーチャー（業界適応が必要な層） | 拡張 1 で汎用エージェントテンプレート化が可能と確認される |
| 拡張 3 | 非エンジニア業務人材一般（北極星） | 拡張 2 で UI / テンプレートが十分に抽象化済みと判断できる |

### ペルソナ成功判定基準

本 ADR に基づく MVP 実装後、以下が成立すればペルソナ定義は妥当と判断する：

1. プライマリペルソナの JTBD シナリオを、MVP 機能だけで **30 分以内**に完走できる
2. ドッグフーディングで、セカンダリペルソナ（PM / エンジニア自身）が週 1 回以上実利用する
3. コア UC 3 本のうち少なくとも 1 本で、手動比 **50% 以上**の時間削減が観測される

## Consequences

### ポジティブ

- ペルソナ選定基準（3 軸）の明文化により、#9 MVP スコープ・#11 アーキテクチャで判断軸が揃う
- JTBD 形式の具体シーンにより、MVP 機能の取捨選択が定量基準で判断できる
- ビーチヘッド戦略の明示で、北極星「誰もが」との距離感を**意図的な管理対象**として扱える
- ADR-0003 の残留リスク（ドッグフーディング困難）に対して、セカンダリペルソナを介した具体策を提示

### ネガティブ / リスク

- プライマリ 1 職種に絞るため、想定外ユーザー層のインサイトを取り逃すリスク
  → 対策: 拡張ロードマップの移行条件を明示し、段階的に広げる
- JTBD シーンが 1 本に寄っているため、検証結果がシーン固有の特性に引きずられる可能性
  → 対策: #9 MVP スコープ確定時に成功判定基準を多角化する
- コア UC 3 本はいずれも並列エージェント構造を要求するため、技術難度が相対的に高い
  → 対策: #10 技術スタック選定で並列実行・オーケストレーション機構を重点評価

### 中立 / 開発プロセス上のドッグフーディング計画

- **ADR・設計ドキュメントの草稿生成**は、本プロジェクト開発中のドッグフーディング施策として位置付ける（ユーザー向けユースケースには含めない）
- 本プロジェクトチーム（PM + エンジニア）自身がセカンダリペルソナを演じ、各開発フェーズで実利用フィードバックを収集する
- FAQ 整理は単発 LLM で十分代替可能なため、MVP コア UC から除外した（将来拡張での再評価は可能）
