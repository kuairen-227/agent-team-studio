# 0006. 開発プロセスの軽量アジャイル化

## Status

accepted

- 作成日: 2026-04-17
- 関連: Issue #28

## Context

現在の開発プロセスは7段階のフェーズラベル（`phase:planning` 〜 `phase:operations`）と docs/README.md のフェーズマップにより、ウォーターフォール的な逐次進行を前提としている。

1人エンジニア + AI 協働の体制では、企画・要件定義・設計が同時並行で進むのが実態であり、フェーズ管理は形式的なオーバーヘッドになっている。また `requirement` ラベルは `enhancement` と用途が重複しており、分類の判断コストが無駄に発生している。

## Considered Alternatives

### フェーズ管理の代替

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | フェーズラベルを維持 | 却下 — 実態と乖離しており形式的な負担になっている |
| B | フェーズラベルを廃止し、ステータスラベルに移行 | **採用** — Issue の現在の状態を表すほうが実用的 |
| C | ラベルなし（マイルストーンのみで管理） | 却下 — Issue 一覧でのフィルタリングが不便になる |

### ステータスラベルの粒度

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | 3段階（todo / in-progress / done） | 却下 — blocked や review を区別できない |
| B | 5段階（backlog / ready / in-progress / blocked / review） | **採用** — 必要十分な粒度 |
| C | GitHub Projects のステータスに委譲 | 却下 — 現時点では Projects を使っていない |

## Decision

### フェーズラベルの廃止

`phase:*` ラベル7つを削除し、以下のステータスラベルに置き換える：

| ラベル | 用途 |
| --- | --- |
| `status:backlog` | 未着手・優先度未定 |
| `status:ready` | 着手可能 |
| `status:in-progress` | 作業中 |
| `status:blocked` | ブロック中 |
| `status:review` | レビュー待ち |

### `requirement` ラベルの廃止

`enhancement` と重複するため削除する。Feature Request テンプレートの自動付与からも除外する。

### マイルストーン運用の柔軟化

「全 Issue クローズでマイルストーンクローズ」の厳密なルールを撤廃し、区切りのよいタイミングでクローズする運用に変更する。

### ドキュメントの簡素化

docs/README.md のフェーズマップ（開発ライフサイクルとの対応表）を削除し、ディレクトリの役割説明のみに簡素化する。

## Consequences

- フェーズの逐次進行を前提としないため、企画・設計・実装を柔軟に行き来できる
- ステータスラベルにより Issue の現在の状態が一目でわかる
- `requirement` と `enhancement` の使い分け判断が不要になる
- フェーズベースの進捗管理はできなくなるが、1人チームでは不要
- ADR も複数を並行検討する運用が自然になる
