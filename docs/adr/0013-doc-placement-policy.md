# 0013. ドキュメント配置ポリシー

## Status

accepted

- 作成日: 2026-04-21
- 関連: ADR-0007（AI 駆動開発アーキテクチャ）

## Context

`docs/product/` と `docs/design/` の役割は「What/Why vs How」として定義されているが、具体的な配置基準がなく、技術要素が product/ に混入しやすい。特にテンプレート仕様のシステムプロンプト内に JSON 出力スキーマがインライン定義され、design/ 側と重複する問題が発生していた。

また、本プロジェクトは AI 駆動開発を前提としており（ADR-0007）、ドキュメントは人間だけでなく AI の実装入力としても機能する。アジャイル・リーンの「コードで表現できることはコードを唯一のソースとする」原則と、AI がドキュメントを読んで実装する開発スタイルの両立が求められる。

## Considered Alternatives

### 境界の引き方

| # | 選択肢 | 判定 |
| --- | --- | --- |
| A | 読み手テスト + コードが SSoT 原則 | **採用** — 読み手で配置先を判断し、実装後はコードを SSoT とする |
| B | 厳密な技術用語ブラックリスト | 却下 — LLM プロダクトでは技術用語と業務用語の境界が曖昧。機械的な判定は誤検知が多い |
| C | 現状維持（抽象的な What/How 指針のみ） | 却下 — 既に配置の迷いと重複が発生している |

### システムプロンプト内の出力スキーマ

| # | 選択肢 | 判定 |
| --- | --- | --- |
| A | 自然言語概要 + design/ 参照に置換 | **採用** — 重複を排除し design/ を SSoT にする。実行時にスキーマを展開する旨を注記 |
| B | 重複を許容 | 却下 — 変更時の同期漏れリスクが高い |
| C | プロンプト自体を design/ に移動 | 却下 — プロンプトはエージェントの振る舞い仕様であり、product/ の領分 |

## Decision

### 原則 1: 読み手テスト

| | product/ | design/ |
| --- | --- | --- |
| 読み手 | PO・PM・ビジネス | エンジニア・アーキテクト |
| 問い | What / Why | How |
| 含む | 意図・振る舞い・UX・ビジネスルール | データ契約・型・アーキテクチャ・パラメータ |

迷ったときの判断基準:

1. 非技術者が読む必要があるか → product/
2. エンジニアが実装に使うか → design/
3. データ型を定義しているか（JSON Schema, enum リテラル, 型制約）→ 常に design/

### 原則 2: コードが SSoT（フェーズ別適用）

| フェーズ | SSoT | ドキュメントの役割 |
| --- | --- | --- |
| 設計段階（コード未存在） | ドキュメント（暫定 SSoT） | AI・人間への実装入力仕様 |
| 実装後 | コード（型・スキーマ・テスト） | コードへの参照 + コードで表現できない意図・背景 |

- design/ のスキーマ定義は実装前の設計図。実装後はコード参照に移行する
- product/ の価値は不変。ユーザー意図・振る舞い・UX はコードで表現しにくい
- ADR の価値も不変。判断理由はコードに残らない

### 適用: システムプロンプト内の出力フォーマット

- プロンプト本文は product/ に配置（エージェントの振る舞い仕様）
- プロンプト内の出力フォーマット（JSON 構造）は自然言語概要 + design/ スキーマへの参照に置き換える
- design/ の I/O スキーマが Single Source of Truth
- 実行時にはスキーマ定義をプロンプトに展開する旨を注記する

## Consequences

- 各 README に配置ガイドラインを追記する必要がある
- 既存の `docs/product/templates/competitor-analysis.md` のインライン JSON スキーマを自然言語+参照に修正する必要がある
- 実装フェーズ移行時に design/ ドキュメントをコード参照に更新する運用が発生する
- AI（Claude Code）がドキュメント作成時にガイドラインを参照できるよう、スキルを整備する
