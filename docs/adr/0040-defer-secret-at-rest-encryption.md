# 0040. シークレット at-rest 暗号化の見送りと残存リスクの運用統制による受容

## Status

accepted

- 作成日: 2026-06-14
- 関連: [ADR-0039](./0039-secret-read-guard.md)（再評価元：論点 2 / Decision 3「dotenvx 採用方向」を本 ADR で取り下げ）, [ADR-0037](./0037-ai-execution-sandbox-policy.md)（前提：permissions.deny + egress firewall の多層防御）, Issue #292

## Context

[ADR-0039](./0039-secret-read-guard.md) は Scope A（Read deny の変種拡張 ＋ PreToolUse(Bash) ガードフック）を採用し、シークレット読取・環境変数ダンプの経路を多層で遮断した。同 ADR は残る穴 3（平文 `.env` がディスク上に存在）の解消方針として **dotenvx による at-rest 暗号化を採用方向**とし、「実導入時に別 ADR（ADR-0040 を想定）で拡張する」と予告していた（Scope B）。

実導入の前に Scope B を再評価したところ、本プロジェクトの文脈では at-rest 暗号化の正味の増分が小さいことが判明した。

- **トポロジー**: `docker-compose.yml` で Claude Code はアプリと**同一 `app` コンテナ・同一 `node` ユーザー**で稼働する（`command: sleep infinity`、workspace を bind-mount）。アプリ起動に必要な復号鍵は同一環境に存在せざるを得ず、Claude はアプリを起動できる＝復号もできる。したがって OS 権限・コンテナ分離による読取遮断（王道）は前提が成立せず、どの at-rest 暗号化も「ガードに穴があった時に漏れるのが平文でなく暗号文」という**二次的 defense-in-depth にとどまる**。
- **既存対策との冗長**: dotenvx の主価値「暗号化済み `.env` を git に commit して露出を防ぐ」は、本リポジトリが `**/.env` を `.gitignore` ＋ secretlint で既に塞いでいるため冗長。残る価値は鍵 1 点への集約と上記の二次層のみ。
- **実コスト**: 実 `.env` の暗号化・docker-compose の復号配線は実シークレットを伴うためサンドボックスで検証できず、ローカル限定作業になる。
- **現実的な最大損失**: 本プロジェクト（個人・学習スコープ、LLM API キーが本命シークレット）で現実的に最も痛いのは、データ漏洩より**請求の暴走（bill shock）** — キー流出や無人ループ暴走によるトークン消費。これは暗号化では止まらない。

## Considered Alternatives

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | **dotenvx を ADR-0039 の予定どおり実導入**（at-rest 暗号化 + `dotenvx run` 注入） | 却下 — 主価値（git 露出防止）が既存 gitignore+secretlint と冗長。トポロジー上は二次層にすぎず、実導入は実シークレットを伴いローカル限定でコスト高 |
| B | SOPS + age（暗号化 at-rest + 非対話復号） | 却下 — dotenvx と同クラスの効果で、追加依存と compose 連携の自作コストが上乗せ |
| C | ストア型（Infisical / Doppler / Bitwarden / 1Password） | 却下 — 2026 年のベストプラクティス本命（ディスクから排除しランタイム注入）だが、アカウント / サーバ必須で「無料・ローカル完結・アカウント不要」の制約外。self-host は学習スコープ過大 |
| D | OS キーチェーン型（getcloak / CloakEnv 等） | 却下 — TTY / D-Bus / 対話承認を要し、ヘッドレス DevContainer に不適（ADR-0039 でも非対話不可として整理済み） |
| E | **at-rest 暗号化を見送り、残存リスクを運用統制で受容** | **採用** — トポロジーと既存対策により暗号化の増分が小さい一方、運用統制（支出上限・専用キー・イベント駆動ローテ）は現実的最大損失（bill shock）を低コストで頭打ちにできる |

## Decision

1. **at-rest 暗号化（dotenvx を含む）の実導入を見送る。** [ADR-0039](./0039-secret-read-guard.md) 論点 2 / Decision 3 の「dotenvx 採用方向」を本 ADR で取り下げ、穴 3（平文 `.env` のディスク存在）は**受容する残存リスク**として明示する。Scope A（読取ガード）は ADR-0039 のまま有効で、本 ADR は ADR-0039 を supersede しない。

2. **受容条件として以下の運用統制を敷く**（漏洩・暴走の被害を限定する。予防ではなく blast radius / 窓の限定）:
   - **支出上限 + 使用量アラート**: Anthropic Console のワークスペース単位の支出上限と使用量通知を設定し、流出・暴走時の金銭損失を頭打ちにする。一度きりの設定で toil ゼロ。最優先。
   - **専用 API キーの分離**: 本プロジェクト専用のキーを 1 本に分け、他用途・本番と混在させない。流出時の影響範囲をこの 1 本に閉じ、即時無効化を容易にする。
   - **イベント駆動 + 低頻度定期のローテーション**: 「漏洩・誤用を疑ったら即無効化」を主とし、定期ローテは 3〜6 か月程度の軽いものにとどめる。高頻度の機械的ローテは個人運用では toil が勝つため採らない。

3. **主たる保護は Scope A（[ADR-0039](./0039-secret-read-guard.md)）と egress allowlist firewall（[ADR-0037](./0037-ai-execution-sandbox-policy.md)）が担う。** 本 ADR の運用統制はそれらの上に乗る別レイヤー（被害限定の運用統制）である。

4. **再評価トリガを定める。** 無人・長時間の自律ループ（#289–291）が常態化し at-rest 暗号化の実需が顕在化した場合、または本プロジェクトを共有環境・複数ユーザーへ拡張する場合は、本見送り判断を再評価する。

## Consequences

### ポジティブ

- 実装・運用コストをほぼかけずに、現実的な最大損失（bill shock）を支出上限で頭打ちにできる。
- 保護対象が「専用キー 1 本」に明確化し、Scope A が守る対象が単純になる。
- YAGNI / MVP 整合：実需が出るまで複雑なシークレット基盤を持ち込まない。

### ネガティブ / リスク

- 平文 `.env` がディスクに残り続ける（穴 3 は未解消）。Scope A ガードに将来穴が空いた場合、漏れるのは暗号文でなく平文になる。→ 対策: Scope A の多層ガード（[ADR-0039](./0039-secret-read-guard.md)）＋ egress allowlist firewall（[ADR-0037](./0037-ai-execution-sandbox-policy.md)）＋ 専用キーによる影響範囲の限定＋疑い時の即時無効化運用で多重に backstop する。
- 運用統制（支出上限設定・ローテーション）は人手に依存する。→ 対策: 一度きりで済む設定（支出上限・専用キー分離）を主軸に置き、ローテーションはイベント駆動とすることで継続的な toil を最小化する。

### 中立

- [ADR-0039](./0039-secret-read-guard.md) が示した dotenvx 導入 runbook（`docs/guides/env.md`）は、将来の再評価に備えた参照として残すが、位置づけを「予定」から「保留（再評価条件付き）」へ変更する。
