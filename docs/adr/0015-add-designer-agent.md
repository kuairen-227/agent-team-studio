# 0015. デザイナーエージェントの追加

## Status

accepted

- 作成日: 2026-04-26
- 関連: ADR-0011（前提）, Issue #80

## Context

ADR-0011 で PO / PM / Architect / QA の 4 エージェントが定義されたが、UI/UX とブランドの専門領域はカバーされていなかった。Issue #80 のフォローアップで [docs/design/ui-patterns.md](../design/ui-patterns.md) と [docs/product/brand.md](../product/brand.md) が独立したドキュメントとして整理され、これらを立脚点とするレビュー観点が既存 4 エージェントの範囲外として明確になった。

具体的に既存エージェントでカバーできない観点:

- ブランド軸（誠実さ・判断材料の提供）が UI 文言・状態表示・コンポーネント選定に反映されているか
- 業界標準（Nielsen 10 ヒューリスティック / Laws of UX / WCAG POUR）と判断軸が整合しているか
- 状態パターン（loading / empty / error / partial-failure）が ui-patterns.md §3 に従っているか
- ボイス&トーン規約が UI 文言・API メッセージで一貫しているか
- アクセシビリティ最低限（キーボード・aria-live・focus 管理・コントラスト）が満たされているか

加えて、PO がプロダクト判断（要件・受入条件・スコープ）を評価する際にも brand 軸との整合性は必要だが、これまで PO の参照範囲に brand.md が含まれていなかった。

## Considered Alternatives

### エージェント追加の必要性

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | designer エージェントを追加せず、既存の architect / qa にカバーさせる | 却下 — architect は構造設計、qa はテスト・コード品質が専門領域。UX/ブランドは独立した知識領域 |
| B | UX とブランドで別エージェントに分割（uxdesigner + branddesigner） | 却下 — MVP 規模では分割の必要性が薄い。両者は共通の参照ドキュメント（ui-patterns.md / brand.md）を持つ |
| C | designer 1 つにまとめて UX とブランドの両方を担う | **採用** — ADR-0011 と同じ「専門知識の領域」原則に沿う |

### ブランド観点の責務分担

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | designer のみ brand.md を参照する | 却下 — 要件レベル（プロダクト判断）でも brand 軸との整合性チェックが必要。designer は実装視点のため上流をカバーしきれない |
| B | po + designer の両方が brand.md を参照する（責務分担を明示） | **採用** — po: プロダクト判断と brand 軸の整合 / designer: UI/UX 実装と brand 軸の反映、で重複なく分業 |
| C | architect / qa にも brand.md を参照させる | 却下 — 架構判断・コード品質と brand は直接関係が薄い。必要になった時点で追加 |

## Decision

### エージェント追加（ADR-0011 のテーブルを拡張）

| エージェント | 専門領域 | 主な関心事 |
| ------------ | --------- | ---------- |
| `designer.md`（新規） | UX / ブランド | ブランド軸整合・状態パターン・コンポーネント選定・WCAG POUR・ボイス&トーン |

専門知識として Nielsen 10 ヒューリスティック / Laws of UX / WCAG 2.1 POUR を保持し、プロジェクト固有の判断軸・ブランド軸は ui-patterns.md / brand.md への参照に留める（既存エージェントの「ドキュメント参照スタイル」に準拠）。

### レビュー時のエージェント組合せ追加

| レビュー対象 | エージェント | 根拠 |
| ------------ | ----------- | ------ |
| UX（UI 設計、画面フロー、ブランド・ボイス、UI 実装の細部） | `designer` | UX 整合・ブランド軸の UI 反映・状態パターン・アクセシビリティ |

UI コンポーネントの実装レビューが必要な場合は `code` レビュー（architect + qa）と `ux` レビュー（designer）を併用する。

### ブランド参照の責務分担

`po` と `designer` の両方が brand.md を参照する。重複を避けるため責務を明示する:

- **po**: プロダクト判断（要件・受入条件・スコープ）が brand 軸（誠実さ・判断材料の提供）と矛盾しないかを判定する
- **designer**: UI/UX 実装（文言・状態表示・コンポーネント選定）が brand 軸とボイス&トーン規約を反映しているかを判定する

`architect` / `qa` への brand.md 参照追加は当面行わない（架構判断・コード品質と brand の直接関係が薄いため）。

### スキル定義の更新

`/review` スキルに以下を追加:

- レビュー種別判定: `ux ...` を新設
- 対象推測: `docs/design/ui-patterns.md` / `docs/product/brand.md` / `docs/product/screen-flow.md` / `apps/web/src/...` の React コンポーネントを UX レビューとして推測
- エージェント委譲: UX レビュー → `designer`
- レビュー粒度ガイドライン: UX レビューの焦点（ブランド軸・状態パターン・アクセシビリティ・ボイス&トーン）と除外（バックエンド設計・データ構造・API のエラーコード設計）を明記

## Consequences

- UI/UX とブランドのレビューが、他のレビューと同じ `/review` スキルから一貫したパターンで呼び出せるようになる
- プロダクトレビュー時にも brand 軸との整合がチェックされ、要件レベルの矛盾を早期に発見できる
- designer のドキュメント参照スタイルにより、ui-patterns.md / brand.md の更新がレビュー観点に自動的に反映される
- ADR-0011 のエージェント定義テーブルとレビュー組合せテーブルが本 ADR で拡張される（ADR-0011 自体は accepted のまま）
- 将来 UI コンポーネントが増えた時に designer の評価基準を拡張する余地が残る
- セキュリティ専門エージェントなど、他のエージェント追加時も同じパターンで拡張可能
