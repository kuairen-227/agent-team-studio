# 0028. コード可読性原則ドキュメントの新設

## Status

accepted

- 作成日: 2026-05-07
- 関連: ADR-0017（設計・開発原則の新設）, ADR-0019（主題別分割の運用ルール）

## Context

`docs/principles/README.md` には 6 軸の横断原則（本質志向・削減主義、節度など）があるが、コードそのものの可読性に特化したガイドラインが存在しなかった。

- 文体規約（§6）はテキスト出力を対象とするが、コードの命名・関数サイズ・ネスト・コメントスタイルは対象外
- `architect.md` は構造・型安全性を評価するが「読みやすさ」は評価軸に含まれていなかった
- 原則間が競合したとき（DRY vs 可読性 等）のタイブレーカーが未定義だった

## Considered Alternatives

### A. principles/README.md に追記する

- メリット: ファイルを増やさない
- デメリット: README が肥大化する。ADR-0019 の分割トリガ（独立した主題・十分な分量）を既に満たしており、分割が適切

### B. docs/principles/coding.md を新設する（採用）

- testing.md と対称的な構造になる
- 命名・関数サイズ・ネスト・コメント・型・複雑度の 6 軸で独立した主題を成立させる
- ADR-0019 §3 の分割トリガ判定: 独立性あり（コード可読性はテスト・設計と主題が分離）、分量あり（6 サブセクション + アンチパターン表）

## Decision

`docs/principles/coding.md` を新設し、コード可読性の判定軸 SSoT とする。

原則間の競合は数値順位リストではなく、既知の競合パターンへの解決方針として定義する。ほとんどの場合は競合しないため、ケース別方針の方が実用的。

| 競合 | 解決方針 |
| --- | --- |
| DRY vs 可読性 | 抽象化が名前で意図を語れなければ可読性を優先する |
| 型安全性 vs 可読性 | 型安全性を優先する。ただし型名自体が可読性を高める設計を先に試みる |
| セキュリティ vs 可読性 | セキュリティを優先する。可読性の犠牲が大きい場合はコメントで理由を残す |

`.claude/agents/architect.md` の評価基準に可読性チェックを追加する。コードレビュー時に architect が `coding.md` を参照する経路を作ることで、既存の伝播構造（agents が principles を参照）がそのまま機能する。

## Consequences

- architect エージェントが可読性（命名・ネスト・関数サイズ・コメント）を評価軸として持つ
- DRY vs 可読性 等の原則競合が発生した際の解決方針が明文化される
- `review` / `implement-feature` スキルは変更不要（architect 経由で間接的に反映される）
- コード例を principles ドキュメントに含めることで、記述が具体になる一方、将来の言語・スタイル変更時に更新コストが生じる
