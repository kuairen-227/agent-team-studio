# 0011. ロールベースエージェントアーキテクチャ

## Status

accepted

- 作成日: 2026-04-20
- 関連: ADR-0007（前提）, ADR-0006（前提）

## Context

ADR-0007 で「ハイブリッド方式（スキル + ロール定義）」が採択され、ロールの想定（PM補佐、アーキテクト、実装者、QA、レビュアー）が定義された。しかしロール定義の具体的な実装方法は未決定のままだった。

開発フェーズに入り、以下の 2 つの活動で専門的な視点が必要になった:

1. **タスク管理**: PM の知識に基づく進捗管理・リスク分析
2. **レビュー**: 対象に応じた専門知識による成果物チェック

ロール知識の配置先と、スキルとの連携方法を決める必要がある。

### エージェント設計の原則

検討の過程で、エージェントは「行為」ではなく「専門知識の領域」として定義すべきという原則が明確になった。当初検討した `reviewer`（レビュアー）エージェントは「レビューという行為」を定義しており、他のエージェント（architect, qa）と性質が異なっていた。レビューは専門家を組み合わせる行為であり、専門領域ではない。

また、ADR-0007 の「PM 補佐」ロールにはプロジェクト管理（PM）とプロダクト管理（PO）の 2 つの異なる専門領域が混在していた。

## Considered Alternatives

### ロール知識の配置先

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | `.claude/roles/`（カスタムディレクトリ） | 却下 — 公式サポート外。スキルから `!cat` で読み込む非標準パターンになる |
| B | `.claude/agents/`（エージェント定義） | **採用** — 公式サポート済み。ツール制限・モデル指定が可能で、スキルから委譲できる |
| C | スキル内に直接記述 | 却下 — 複数スキルで同じロール知識を使う場合に重複する |
| D | `.claude/rules/`（ルール定義） | 却下 — ルールはセッション開始時やファイルパスマッチで自動読み込みされる。オンデマンドで特定ロールを適用する用途に合わない |

### スキル構成

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | ロールごとにスキルを作成（review-code, review-design 等） | 却下 — ADR-0007 で却下済みの「スキル拡張方式」と同じ問題（スキル数膨張） |
| B | 活動ごとに汎用スキルを作成（manage-task, review） | **採用** — スキル数を抑えつつ、対象に応じてエージェントを組み合わせる |

### エージェント構成

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | ADR-0007 の 5 ロールをそのままエージェント化（PM, アーキテクト, 実装者, QA, レビュアー） | 却下 — レビュアーは「行為」であり専門領域ではない。PM に PO が混在 |
| B | 専門領域で再分類（PO, PM, アーキテクト, QA） | **採用** — 各エージェントが独立した知識領域を持ち、レビュー時に組み合わせる |

## Decision

### エージェント定義（`.claude/agents/`）

エージェントは「専門知識の領域」として定義する。各エージェントは視点・評価基準・チェックリスト・アンチパターンを含む。

| エージェント | 専門領域 | 主な関心事 |
| ------------ | --------- | ---------- |
| `po.md` | プロダクト | ユーザー価値、JTBD、受入条件、スコープ |
| `pm.md` | プロジェクト管理 | 進捗、リスク、依存関係、スケジュール |
| `architect.md` | 構造設計 | アーキテクチャ、ADR整合性、コード構造、型安全性 |
| `qa.md` | 品質保証 | テスト戦略、エラー処理、セキュリティ、境界値 |

`implementer`（実装者）は独立エージェントとしない。既存の `process-issue` スキルが実装者ロールを担う。

### レビュー時のエージェント組合せ

レビューは「行為」であり、対象に応じて適切な専門家エージェントを組み合わせる。

| レビュー対象 | エージェント | 根拠 |
| ------------ | ----------- | ------ |
| プロダクト（要件、Issue） | `po` + `pm` | po: 正しいものを作ろうとしているか / pm: 実現可能か |
| 設計（ADR、アーキテクチャ） | `architect` | 構造的妥当性と一貫性 |
| 実装（コード、PR） | `architect` + `qa` | architect: 構造面 / qa: 品質・安全性面 |
| テスト | `qa` | テスト設計と品質 |

### スキル定義

| スキル | 委譲先 | 方式 |
| ------- | ------- | ------ |
| `/manage-task` | `pm` | `context: fork agent: pm` で静的委譲 |
| `/review` | 対象に応じて動的選択 | 引数から対象を判定し、Agent ツールで適切なエージェントに委譲 |

### 付随する修正

`create-issue` スキルから ADR-0006 で廃止済みの `phase:*` ラベル参照と `requirement` ラベル参照を削除する。

## Consequences

- エージェントが「専門知識の領域」として一貫した設計になる
- レビューは専門家の組合せとして表現され、対象の変化に柔軟に対応できる
- PO と PM を分離したことで、プロダクト判断とプロジェクト管理の視点が混同されなくなる
- スキル追加は 2 つ（manage-task, review）に留まり、スキル膨張を防げる
- エージェント定義は他のスキルからも再利用可能
- エージェントの知識は初期版であり、運用を通じて評価基準・チェックリストを改善する必要がある
- 将来的にエージェントを細分化する可能性がある（例: セキュリティを QA から分離）
