# 0046. vN.x マイルストーンをフェーズ期の後続枠として一般化し、v2.1 をデザイン質感改善に割り当てる

## Status

accepted

- 作成日: 2026-06-27
- 関連: [ADR-0031](./0031-version-naming-convention.md)（前提：バージョン命名規約を拡張）, [ADR-0044](./0044-v2-scope-definition.md)（v2.1 予約を一部再評価）, Issue #328（デザイン質感改善）, #320（v2.0 親トラッカ）

## Context

ADR-0031 はプロダクトのバージョンを「機能フェーズ」に紐づけて定義した（v1.0 MVP / v1.x MVP 改善・リファインメント / v2.0 機能拡張 / v3.0 ...）。このうち vN.x（マイナー）で明文化されているのは v1.x（=「MVP 改善・リファインメント」）のみで、**v2.x 以降の意味は未定義**だった。

v2.0（機能拡張、ADR-0044）の進行中に、機能ロードマップに属さない横断テーマ「デザイン質感改善」（Issue #328）が発生した。これは brand.md §3「独自 VI は作らない」を維持したまま結果マトリクス画面の見た目の質感を引き上げる、リファインメント性格の作業で、v2.0 のスコープ（ADR-0044 軸1=基準3 決着 / 軸2=UC#2 汎用性検証）には含まれない。これを既存のバージョン体系のどこに置くかが問題になった:

- **v2.0 に混ぜる** → ADR-0044 が過大スコープ回避のため軸を絞った狙いと衝突
- **非バージョンのテーマ型マイルストーン** → ADR-0031 の「マイルストーン＝バージョン」前提から外れる
- **v1.x に戻す** → v2.0 進行中に番号が前後し、かつ v1 期（MVP）の作業ではない

加えて、ADR-0044 §3 は UC#2 を No-Go 時に分離する先として「v2.1 - Feature Expansion」を条件付きで予約していた。v2.x をデザインに用いるなら、この予約との衝突を解消する必要がある。

## Considered Alternatives

### 論点 1: v2.x の意味づけ

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | vN.x を「vN 期の後続マイナー増分（改善・リファインメント・後送り機能の受け皿）」として一般化し、v1.x をその v1 期特殊化として包含 | **採用** — ADR-0031 の既存定義（v1.x）と矛盾せず、未定義だった v2.x 以降に一貫した意味を与える。テーマ型の例外を作らずにバージョン体系へ吸収できる |
| B | デザインを非バージョンのテーマ型マイルストーン（例: Design Quality）に置く | 却下 — ADR-0031 の「マイルストーン＝バージョン」前提に例外を持ち込み、以後どこまでテーマ型を許すかの線引きが恒久的に必要になる |
| C | デザインを v1.x（例 v1.3）に置く | 却下 — v2.0 進行中に番号が前後し参照が混乱する。対象画面は MVP 由来だが作業自体は v2 期の発生で、v1 期（MVP リファインメント）の括りに収まらない |

### 論点 2: v2.1 の割当先（ADR-0044 予約との衝突）

| # | 選択肢 | 判定 |
| - | --- | --- |
| D | v2.1 = デザイン質感改善、UC#2 分離先を v2.2 へ繰下げ | **採用** — デザイン質感改善（#328）は before/after 検証済みで即着手可能な確定作業。一方 UC#2 分離は軸1 完了後の Go/No-Go 条件付きで未確定。確定済みの作業に先行番号を与え、条件付きの作業を後続へ回すのが着手順と整合する |
| E | v2.1 = UC#2 分離（ADR-0044 予約を維持）、デザインを v2.2 | 却下 — 条件付き・未着手の UC#2 に先行番号を占有させ、即着手の #328 を後ろへ回す。番号と着手順が逆転する |

## Decision

1. ADR-0031 を拡張し、vN.x（x≥1）を「**vN.0 リリース後、次のメジャーフェーズ着手前までに発生する後続マイナー増分（改善・リファインメント、および後送り／分離された機能スコープ）を束ねる枠**」と定義する。ADR-0031 が定めた v1.x（MVP 改善・リファインメント）は、本定義の v1 期における特殊化として包含され、変更しない。
2. **v2.1 - Design Refinement** をデザイン質感改善（Issue #328 起点）に割り当てる。
3. ADR-0044 §3 が UC#2 No-Go 時の分離先として予約していた「v2.1 - Feature Expansion」を **v2.2 - Feature Expansion** へ繰り下げる。ADR-0044 のこの一点のみを再評価し、他のスコープ判断（軸1/軸2 の内容・Go/No-Go 基準）は変更しない。
4. 適用: 本決定以降の新規マイルストーン名は本定義に従う。ADR-0031 の既存記述・既存マイルストーン名は変更しない。ADR-0031 の既存定義（バージョン表・適用範囲）自体は不変で、未定義だった v2.x 以降の領域を補完するのみであるため、supersede には該当せず拡張に留める。

## Consequences

### ポジティブ

- 未定義だった v2.x 以降のマイナーバージョンに一貫した意味が与えられ、機能ロードマップに属さない横断テーマ（質感改善等）もバージョン体系内に収まる
- テーマ型マイルストーンという例外を導入せずに済み、ADR-0031 の「マイルストーン＝バージョン」前提を保てる
- 確定作業（#328）に先行番号、条件付き作業（UC#2 分離）に後続番号、という着手順と整合した番号付けになる

### ネガティブ / リスク

- ADR-0044 の予約番号（v2.1）を動かすため、#320 など v2.1 を参照する記述との不整合が生じうる → 対策: 本 ADR と同 PR で ADR-0044 の Status に再評価注記を加え、#320 のコメントで繰下げを周知する
- vN.x の定義を「リファインメント＋後送り機能」と広げたため、v2.x に性格の異なる作業（質感改善と UC#2 機能）が混在しうる → 対策: マイルストーン名（Design Refinement / Feature Expansion）で性格を区別し、混同を防ぐ。今回は v2.1（Design Refinement）と v2.2（Feature Expansion）を別マイルストーンに分離したため同一マイルストーン内の混在は発生しておらず、同一 vN.x 内に混在が生じた場合の規則化は v3.x 定義時に検討する

### 中立

- 本規約は学習プロジェクトの内部運用ルールであり、外部向け SemVer の約束ではない（ADR-0031 の立場を継承）
- マイルストーンの GitHub 上での作成・番号採番は本 ADR の範囲外（運用操作）
