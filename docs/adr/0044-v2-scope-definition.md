# 0044. v2.0 機能拡張のスコープ定義

## Status

accepted（§3 の v2.1 割当は [ADR-0046](./0046-minor-version-milestone-generalization.md) で一部再評価: UC#2 分離先を v2.2 へ繰下げ）

- 作成日: 2026-06-26
- 関連: [ADR-0005](./0005-mvp-scope.md)（前提：v2 ロードマップ）, [ADR-0031](./0031-version-naming-convention.md)（前提：バージョン体系）, [ADR-0038](./0038-autonomous-agent-loop-adoption.md)（前提：フェーズ制 + Go/No-Go の先例）, [ADR-0046](./0046-minor-version-milestone-generalization.md)（v2.1 割当を一部再評価）, Issue #320（親トラッカ）, Issue #321（本 ADR）

## Context

v1.0 MVP は ADR-0005 §MVP 成功基準の検証で **CONDITIONAL PASS** に到達した（[validation-summary.md](../milestones/v1-mvp/validation-summary.md)）。定量基準（1: 30 分以内 / 2: 手動比 50% 削減）は PASS、定性基準（3: 出力品質 / 4: 体感）は CONDITIONAL PASS。続く v1.1（製品リファインメント）で基準4 の引き金（待機 UX・入力透明性・エラー画面）を解消し、v1.2（開発プロセス＝自律ループ）も収束した。

ここでプロダクトは ADR-0005 §v2 ロードマップへ移行する段階に入る。次マイルストーン **v2.0 - Feature Expansion** のスコープを確定する必要があるが、確定にあたり 2 つの制約がある:

1. **基準3 の CONDITIONAL が未決着**。引き金は「出典 URL 欠如」と「入力データのファクトチェック未実装」（validation-summary §2 / §4）。v2-priority-memo（`docs/milestones/v1-mvp/v2-priority-memo.md`。本 ADR が用いる `V1〜V10` = MVP リファインメント課題、`N1〜N7` = ADR-0005 §v2-3 新機能のラベルは同メモに準拠）§5 では V2（ファクトチェック）と N2（Web 検索）を「v1.2 でセット着手」と仮置きしていたが、v1.2 が自律ループに振り替わったため宙に浮いている。
2. **MVP で過大スコープの教訓**を得ている（ADR-0005 §UX モデル / Hero UC 1 本への集中）。v2.0 で ADR-0005 §v2 新機能（N1〜N4）を無計画に抱えるとこの教訓に反する。

また、v2-priority-memo §6 の ADR 起票トリガー「ADR-0005 §v2 新機能（N1〜N4）のいずれかを採用する場合」に該当するため、スコープ採用を ADR として残す必要がある。本 ADR は v2.0 の **範囲・順序・完了判定** を定める。Web 検索 API の選定（#322）、UC#2 の設計（軸2）など個別の技術判断は本 ADR のスコープ外で、後続 Issue に委ねる。

## Considered Alternatives

### 論点 1: v2.0 のスコープ範囲

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | 基準3 決着のみ（N2 Web 検索 + V2 ファクトチェック） | 却下 — CONDITIONAL の決着には必要十分だが、ADR-0005 が「v2 で UC 追加時に検証する」とリスク登録した**プラットフォーム汎用性**に踏み込まず、機能拡張マイルストーンとして物足りない |
| B | 基準3 決着 + UC#2 追加（汎用性検証） | **採用** — 残った製品的負債（基準3）を決着させつつ、ADR-0005 §Consequences が明示した唯一の未検証リスク（UC 1 本のみで汎用性未検証）を突く。フェーズ制で過大スコープを抑止する |
| C | N1〜N4 をフル採用 | 却下 — MVP の過大スコープ教訓に反する。N1（AI チーム構成提案）は v2-priority-memo で C 評価（「テンプレート起点が機能と確認済」）、N4（テンプレート編集）も同様に後送り可。一度に抱えると検証の焦点が散る |

### 論点 2: 軸2（UC#2）の組み込み方

| # | 選択肢 | 判定 |
| - | --- | --- |
| D | 軸1・軸2 を v2.0 に固定で同梱 | 却下 — N2 / V2 / UC#2 はいずれも L 規模。3 本を無条件に同一マイルストーンへ固定すると MVP の過大スコープを再演する |
| E | 軸1 を必達とし、軸2 は Go/No-Go ゲートで判断 | **採用** — ADR-0038 の自律ループ（Phase 制 + 各 Phase Go/No-Go）で機能した運用パターンを流用。軸1 完了時点の実測（基準3 PASS・残工数）で軸2 を v2.0 続行 / v2.1 分離に振り分け、スコープ規律を保つ |

## Decision

### 1. v2.0 Hero スコープ（論点 1-B）

v2.0 - Feature Expansion を次の 2 軸に絞る:

- **軸1（基準3 決着）**: N2 Web 検索連携（#323）+ V2 ファクトチェック機構（#324）。MVP 検証で残った基準3 CONDITIONAL を PASS 化する。
- **軸2（汎用性検証）**: UC#2（フィードバック分析）の追加。ADR-0005 が未検証としたプラットフォーム汎用性を初検証する。軸2 は §3 の Go/No-Go を経て起票する。

### 2. 後送り項目（論点 1-C）

- **N1（AI チーム構成提案）**: v2.x 以降。v2-priority-memo C 評価（テンプレート起点が MVP で機能と確認済）を継承。
- **N4（テンプレートのユーザー作成・編集）**: v2.x 以降。同上。
- N5〜N7（UC#3 / 中断・再開 / フィードバックループ）は ADR-0005 §v3 ロードマップのまま据え置く。

### 3. フェーズ順序と Go/No-Go（論点 2-E）

**軸1 → Go/No-Go → 軸2** の順で進める。

- **軸1 完了の定義**: [acceptance-protocol](../milestones/v1-mvp/acceptance-protocol.md) の基準3 観点を再計測し、**CONDITIONAL → PASS** を確認すること。具体的には (a) 調査結果の各観点セルに実在する出典 URL が付与される、(b) 入力参考情報・調査結果の事実矛盾を検出してユーザーに提示できる、の 2 点を満たす。なお (a)/(b) の**具体的な計測シナリオと合否閾値の定義は #323（N2）/ #324（V2）の受入条件に必須事項として課す**。本 ADR は完了の枠組みのみを定め、判定基準の具体化を後続 Issue へ委ねる（判定基準が確定するまで軸1 を完了としない）。
- **軸2 の Go 条件（v2.0 で続行）**: 次をすべて満たす場合。
  1. 軸1 が上記定義で完了している（基準3 PASS）
  2. UC#2 の見込み工数が v2.0 の残容量に収まる
  3. UC#2 が MVP の基本前提（§4）を覆さない範囲で設計可能
- **No-Go（v2.1 へ分離）**: 上記を欠く場合、UC#2 を **v2.1 - Feature Expansion** マイルストーンへ分離し、v2.0 は軸1 で完了とする。

Go 条件 (2) の「v2.0 残容量」の基準点（工数上限の閾値）は本 ADR では数値を固定せず、**軸2 Go/No-Go 判断 Issue の受入条件で定義することを必須とする**（ADR-0038 の「コスト上限の閾値具体化は当該 Phase Issue の必須事項」運用に倣い、判断の丸投げを避ける）。

判断の所在は親トラッカ #320 の Go/No-Go 時点とし、結果（v2.0 続行 / v2.1 分離）を #320 に記録する。

### 4. 維持する MVP 基本前提

v2.0 は ADR-0005 の以下の基本前提を**覆さない**。覆す変更が必要になった場合は別 ADR を起票する（v2-priority-memo §6 トリガー）:

- **一方向実行**（開始→完了。中断・再開は v3 / N6）— V2 ファクトチェックは固定パイプラインへの検証ステージ追加に留め、双方向の協調ロジックは導入しない
- **テンプレート起点のみ**（AI 提案・ユーザー編集は後送り）— UC#2 は新規シードテンプレートの追加であり、ユーザーによるテンプレート編集（N4）は含まない

## Consequences

### ポジティブ

- 基準3 CONDITIONAL の決着が v2.0 軸1 で明確な完了ラインを持ち、MVP 検証で残った唯一の製品的負債を返済できる
- ADR-0005 が未検証としたプラットフォーム汎用性に、Go/No-Go ゲート付きで段階的に踏み込める
- フェーズ制により、軸1 完了時の実測でスコープを再調整でき、MVP の過大スコープ教訓を運用に反映できる
- N1 / N4 を後送りと明文化することで、v2.0 の焦点が「基準3 + 汎用性」に絞られ、後続マイルストーンの起票判断も追跡可能になる

### ネガティブ / リスク

- 軸2 を Go/No-Go まで起票しないため、v2.0 着手時点ではマイルストーンの全体像が軸1 に偏り、外形上のスコープが小さく見える。→ 親トラッカ #320 に軸2 候補と Go/No-Go 基準を明記して補う
- 基準3 の再計測（軸1 完了判定）は n=1 のドッグフーディングに依存し、MVP 検証と同じく統計的有意性は持たない（validation-summary §5）。→ PASS/CONDITIONAL の判定根拠を再計測時に記録し、判断を追跡可能にする
- 軸1 完了の合否を判断するのは開発者自身であり、n=1 に加えて**自己判定のバイアス**を持つ（統計的有意性とは別軸のリスク）。→ 出典付与・矛盾検出の実例を証跡として残し、第三者が追検証できる形で #320 / #323 / #324 に記録する
- 軸1 の N2（Web 検索）と V2（ファクトチェック）は独立した技術領域で、いずれかが長引くと**直線工程の先頭が遅れ、軸2 Go/No-Go のタイミング全体が後退する**。→ N2 / V2 を並行可能な単位に保ち、長期化の兆候を #320 で早期に可視化する
- UC#2 を v2.1 へ分離した場合、汎用性検証が v2.0 内で完結せず、機能拡張の成果が基準3 決着に偏る。→ 分離時は v2.1 を直後マイルストーンとして起票し、間隔を空けない

### 中立

- Web 検索 API の具体選定（#322）、UC#2 のテンプレート/UC 設計は本 ADR のスコープ外。本 ADR は範囲・順序・完了判定のみを定める
- 本 ADR は status=accepted。軸2 Go/No-Go 基準を含むユーザーレビューを経て確定した
