# 0017. 設計・開発原則ドキュメントの新設

## Status

accepted

- 作成日: 2026-05-01
- 関連: ADR-0007（前提）, ADR-0011（前提）, ADR-0013（前提）, ADR-0010（関連）

## Context

ADR-0011 で「エージェント = 専門知識の領域」と定義し、レビューや判断の場面で領域別の評価基準・アンチパターンを `.claude/agents/{po,pm,architect,qa,designer}.md` に集約した。一方で、領域をまたぐ判断軸（YAGNI、本質志向、過剰回避、ユーザー判断優先、文体規約等）は次の場所に散在している:

- `CLAUDE.md`: 文体規約、コミット規約、ユーザー判断優先
- `.claude/agents/*`: 各領域のアンチパターンに、領域固有でない概念（YAGNI 違反、過剰演出、過剰なプロセス等）が独立に記載
- `.claude/skills/*/SKILL.md`: 「判断はユーザーに委ねる」「スコープを守る」が複数スキルで重複定義

直近、`/review` `/resolve-review` に「削減・本質志向」の観点を追加する PR を作成したが、これは特定スキルに閉じる話ではなく、すべての成果物・判断に貫くべき横断軸であることが明らかになり PR をクローズした。

横断軸を一次情報として持ち、各 agent / skill から参照する構造に切り替える必要がある。ADR-0013 の「コードが SSoT」原則と同様に、横断軸にも SSoT を設ける。

## Considered Alternatives

### 配置方針

| # | 選択肢 | 判定 |
| --- | --- | --- |
| A | 各エージェント・スキルに分散維持 | 却下 — 重複と齟齬が生じやすい。実際に PR #90 で表面化した |
| B | `CLAUDE.md` に追記 | 却下 — `CLAUDE.md` はプロジェクト方針・規約・コマンド集として既に役割が確立しており、原則の体系を含めると肥大化する |
| C | `docs/principles/README.md`（単一ファイル）を新設 | **採用** — 横断軸の SSoT を独立して管理できる。最初は単一ファイル、肥大化が観測されたら分割する |
| D | `docs/principles/{topic}.md` で最初から分割 | 却下（初期）— MVP 段階では分割粒度の判断材料が不足。観測してから判断する |

### 含める範囲

| # | 選択肢 | 判定 |
| --- | --- | --- |
| A | 領域横断軸のみ（業界標準の解説は agents/* に委ねる） | 却下 — 「採用している標準は何か」が一覧で見えなくなる |
| B | 横断軸 ＋ 採用する業界標準の帰属表 ＋ 横断的アンチパターン | **採用** — 横断と領域帰属の境界を明示し、重複を避けつつ全体像を見渡せる |
| C | YAGNI/KISS/DRY/SOLID 等を体系的に解説 | 却下 — 領域固有の解釈は agents/* と重複する。学習価値はあるが保守負荷が大きい |

### 領域固有概念の取り扱い

| # | 選択肢 | 判定 |
| --- | --- | --- |
| A | すべての概念を principles に集約し、agents/* は薄くする | 却下 — ADR-0011 の「エージェント=専門知識の領域」と矛盾。領域知識のサイロが崩れる |
| B | 横断軸のみ principles に引き上げ、領域固有具体例は agents/* に残す | **採用** — ADR-0011 の設計を維持し、principles と agents/* で SSoT が分離する |

## Decision

### 1. `docs/principles/README.md` を新設する

`docs/{adr,design,product,guides}/` と並列で `docs/principles/` を設け、最初は単一ファイル `README.md` から開始する。肥大化したら章ごとに分割する判断は将来の ADR で扱う。

### 2. 3 部構成で記述する

- **第 1 部 横断軸**: ユーザー判断優先 / 本質志向・削減主義 / 節度（過剰回避）/ MVP 整合・現実主義 / 誠実さ（判断奪取・失敗隠蔽の禁止）/ 文体規約
- **第 2 部 採用する業界標準（帰属表）**: 標準名と帰属領域を表で示し、定義は重複させない（横断は principles、領域固有は agents/* が SSoT）
- **第 3 部 横断的アンチパターン**: 判断奪取・過剰追加・削減検討の欠落・スコープ越境・形式遵守・好みでの判断・失敗隠蔽

### 3. 領域帰属の分類基準

- 単一領域内でのみ意味を持つ概念（例: モック濫用、shadcn 独自トークン増殖）→ agents/* の領域定義に残す
- 複数領域にまたがる、または成果物全般に適用される概念（例: YAGNI、過剰回避、ユーザー判断優先）→ principles に置く
- 業界標準の定義・体系: 領域帰属させ、principles は帰属表のみで再定義しない

### 4. 伝播方法

- 既存の `CLAUDE.md` / `.claude/agents/*` / `.claude/skills/*` の重複箇所は、principles へのリンク参照に書き換える
- 領域固有具体例は agents/* に残し、横断軸への参照を 1 行追加する
- 新規 agent / skill 追加時は principles を参照する構造にする

### 5. 進め方

ADR（本書）→ `_dev/` で起草 → `docs/principles/README.md` に正式配置 → agents / skills へ伝播、の 3 段階で進める。各段階を独立した PR とする。

## Consequences

- 横断軸の SSoT が確立し、新規エージェント・スキル追加時に参照すべき軸が明確になる
- 領域固有と横断軸の境界判断が新たに必要になる
- `CLAUDE.md` と agents/skills の重複箇所が削減され、文書間の齟齬リスクが下がる
- principles の更新は agents/skills へ伝播する必要があり、伝播漏れのリスクが残る
- 単一ファイルで開始するため、肥大化したときの分割判断は将来時点で必要になる
- 「採用している業界標準」が一覧で見渡せるため、学習目的のプロジェクトとしての可読性が向上する
