# 0019. 設計・開発原則ドキュメントを主題別に分割する（初出: テスト原則）

## Status

accepted

- 作成日: 2026-05-02
- 関連: ADR-0017（前提・部分補足）, ADR-0013（前提）, ADR-0010（関連）, ADR-0011（関連）, Issue #99

## Context

ADR-0017 で `docs/principles/README.md` を横断軸 SSoT として新設し、「最初は単一ファイル、肥大化が観測されたら分割する」と決めた。Issue #99 でテスト原則の整備に着手したところ、求められる内容が以下のように複数主題かつ相応の分量を持つことが判明した:

- 価値あるテストの判定軸（振る舞い vs 実装詳細 / 経路独立性 / 書かない判断）
- 業界標準の体系（AAA / FIRST / Test Pyramid / Test Doubles の Meszaros 分類 / Test Smells）
- 横断的アンチパターン（トートロジー / 過剰モック / カバレッジ追求 等）

これらを README §1〜§3 の単一節として畳み込むと、6 つある横断軸（ユーザー判断優先・本質志向・節度・MVP 整合・誠実さ・文体規約）の見通しが崩れる。一方、横断軸 SSoT という位置付け自体は維持したい。

ADR-0017 §1 が想定する「肥大化したら分割」のトリガが現実に発生したため、分割の運用ルールを本 ADR で確定する。本 ADR は principles を主題別ファイルに分割する初出となるため、後続の分割が同じ枠組みで判断できるよう判断軸を残す。

## Considered Alternatives

### 配置方針

| # | 選択肢 | 判定 |
| --- | --- | --- |
| A | `docs/principles/README.md` に章追加（単一ファイル維持） | 却下 — 6 軸が並ぶ§1 の中にテストだけ突出した分量の節が入り、横断軸の見通しが崩れる |
| B | `docs/principles/testing.md` に分離 | **採用** — 主題が独立しており、README は横断軸の俯瞰を保てる。主題別分割の前例として運用可能 |
| C | ハイブリッド（README に要約軸＋ testing.md に詳細） | 却下 — 同じ概念の SSoT が 2 箇所に分かれ、ADR-0013 の「定義は重複させない」と矛盾しやすい |

### 分割トリガ

| # | 選択肢 | 判定 |
| --- | --- | --- |
| A | 行数のみで判定（例: 100 行超で分割） | 却下 — 機械的すぎ、主題の独立性を捉えられない |
| B | 主題の独立性 ＋ 分量の二条件 | **採用** — 「6 軸の見通しに影響するか」「単独で参照される頻度があるか」で判断する |
| C | 都度 ADR で判断（ルール化しない） | 却下 — 判断ごとに ADR を切るとオーバーヘッドが大きい |

### 分割ファイルの体裁

| # | 選択肢 | 判定 |
| --- | --- | --- |
| A | 主題ごとに自由構成 | 却下 — 横断軸 SSoT 群としての一貫性が崩れる |
| B | README の 3 部構成（横断軸 / 業界標準帰属表 / 横断的アンチパターン）を踏襲 | **採用** — 主題別ファイルでも構造が揃い、読み手の認知コストを抑えられる |

## Decision

### 1. 主題別ファイルへの分割を許容する

`docs/principles/` 配下に `README.md`（横断軸の俯瞰）と主題別ファイル（`testing.md` 等）を並列で置く運用を採用する。Issue #99 では `docs/principles/testing.md` を新設する。

### 2. 分割トリガ

以下の双方を満たす場合に主題別ファイルへ分割する:

- **主題の独立性**: 単独で参照される頻度があり、6 軸の俯瞰に対して突出した節になる
- **分量**: README §1 の 1 軸に収まらず、サブセクションが 3 つ以上必要

両条件を満たさない場合は README 内に節として置く（ADR-0017 §1 の原則を維持）。

### 3. 分割ファイルの体裁

README の 3 部構成（第 1 部 横断軸 / 第 2 部 採用する業界標準（帰属表）/ 第 3 部 横断的アンチパターン）を主題別ファイルでも踏襲する。3 部のいずれかが該当しない場合は省略してよい。

「踏襲」は **3 部の見出し構造を揃える** 意図であり、各部の内容（特に第 2 部の「帰属表」かどうか）は主題側で再定義してよい。例: testing.md 第 2 部は帰属表ではなく業界標準の用語体系として運用する。

### 4. README との整合性

- 主題別ファイルが扱う領域について、README は俯瞰行（1 行リンク）のみを残し、定義の重複を避ける
- README §2 帰属表の参照先には主題別ファイルへのリンクを併記する
- 主題別ファイルの冒頭で `principles/README.md` への戻りリンクを明示する

### 5. 後続の分割

本 ADR の分割トリガを満たす主題が出た場合、ADR を都度切らずに `docs/principles/<topic>.md` を追加してよい。ただし PR 説明で本 ADR を参照し、トリガ充足の根拠を残す。トリガ自体の見直しは別 ADR で扱う。

## Consequences

- principles の主題別分割が運用ルールとして明文化され、後続の分割が同じ判断軸で進められる
- README は横断軸の俯瞰を保ち、主題別の詳細は対応ファイルが SSoT になる
- 主題別ファイルの追加ごとに README の俯瞰行・帰属表参照を更新する責任が発生する。更新漏れの検出は主題別ファイル追加 PR のレビュー観点として確認する（[ADR-0017 §4](./0017-design-development-principles.md) と並び）
- 主題別ファイル間で原則が重複した場合の整理責任が発生する（横断的内容は README、主題固有は主題別ファイル）
- 学習目的のプロジェクトとして、主題別に深掘り可能な構造になり可読性が向上する
- ADR-0017 §1 の「最初は単一ファイル」方針は維持されるが、「肥大化したら分割」の具体運用が本 ADR で確定する
