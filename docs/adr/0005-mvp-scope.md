# 0005. MVP スコープ

## Status

accepted

- 作成日: 2026-04-17
- 関連: ADR-0003（前提）, ADR-0004（前提）, Issue #9

## Context

ADR-0003 でプロダクトコンセプト（非エンジニア向け AI エージェントチーム構築プラットフォーム）、ADR-0004 でターゲットユーザー（プライマリ: 社内新規事業企画担当）と Hero UC 候補（UC #1: 競合調査の並列深掘り）が定まった。

MVP では「特定業務領域において、非エンジニアが AI エージェントチームを使って業務時間を大幅に削減できる」ことを検証可能な形で示す必要がある（ADR-0003 MVP ビジョン）。本 ADR では、最初のリリースに含める機能セットとスコープ外を定義する。

## Considered Alternatives

### Hero UC の範囲

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | UC #1（競合調査）のみ | **採用** — 1 本を深く作り込み、JTBD の定量検証をシンプルに保つ |
| B | UC #1 をコア + UC #2/3 簡易版 | 却下 — 簡易版の中途半端な実装が検証ノイズになる |
| C | UC #1-3 すべてフル実装 | 却下 — MVP として過大。学習プロジェクトとしても焦点が散る |

### UX モデル

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | テンプレート起点のみ | **採用** — 出力品質をコントロールしやすく、非エンジニアにとって分かりやすい |
| B | AI 提案のみ | 却下 — 出力の安定性が不十分で MVP 検証に不向き |
| C | ハイブリッド（テンプレート + AI 提案） | 却下 — MVP として実装コストが過大 |

### エージェント実行の可視化

| # | 選択肢 | 判定 |
| - | --- | --- |
| A | バックエンド完結（結果表示のみ） | 却下 — 「エージェントチームが協働している」価値が伝わらない |
| B | リアルタイム進捗表示あり | **採用** — 非エンジニアの安心感と、チーム協働の可視化を両立。学習目的としてもリアルタイム通信を経験できる |

## Decision

### MVP Hero Use Case

**UC #1: 競合調査の並列深掘り** に絞る。

ADR-0004 プライマリペルソナの JTBD と直結：

> **When** 月次の事業戦略レビュー前、**I want to** 競合 3 社の直近動向を観点別（戦略 / 製品 / 投資 / パートナーシップ）に比較整理したい、**so I can** 役員提案の裏付けを **30 分以内**に固められる。

UC #2（フィードバック分析）、UC #3（論点整理）は MVP スコープ外とし、v2 以降で段階的に追加する。

### MVP 機能一覧

MVP で実装する機能を以下のレイヤーに分けて定義する。

#### 1. エージェントチーム実行基盤

- **テンプレートベースのチーム構成**: 事前定義テンプレートからエージェントチームを起動する仕組み
- **並列エージェント実行**: 観点別（戦略 / 製品 / 投資 / パートナーシップ）に調査エージェントを並列起動
- **統合エージェント**: 各調査エージェントの結果を統合・比較整理するエージェント
- **LLM プロバイダ連携**: 外部 LLM API（Claude API）を利用したエージェント実行

#### 2. Web UI

- **テンプレート選択画面**: 競合調査テンプレートを選択し、パラメータ（競合企業名、観点等）を入力。参考情報（テキスト。URL を貼り付けた場合も Web 取得は行わず、入力文字列をそのまま LLM に渡す）を任意で添付でき、エージェントの調査コンテキストとして利用される
- **リアルタイム進捗表示**: エージェントチームの実行状況をリアルタイムで表示（WebSocket）。MVP の表示粒度はエージェント単位のステータス（待機 / 実行中 / 完了）と、各エージェントの出力のストリーミング表示。エージェント間の依存関係の可視化や思考過程の詳細表示は v2 以降
- **結果表示画面**: 調査結果を観点×競合のマトリクス形式で表示
- **結果エクスポート**: 調査結果を Markdown 形式でコピー / ダウンロード

#### 3. データ管理

- **実行履歴の保存**: 過去の調査実行とその結果を保存・閲覧
- **テンプレート定義の管理**: テンプレートをシステム内で管理（MVP ではシード データとして組み込み、ユーザーによる作成・編集は v2）

### テンプレートの概念モデル

MVP におけるテンプレートは以下を定義する：

- **チーム構成**: 起動するエージェントの役割と数（例: 戦略調査役、製品調査役、統合役）
- **各エージェントのプロンプト**: 役割ごとの指示・観点
- **出力フォーマット**: 最終成果物の構造（例: 観点×競合のマトリクス）
- **入力パラメータのスキーマ**: ユーザーが入力する項目の定義（例: 競合企業名、観点リスト）

テンプレートが定義**しない**もの：

- ツール連携の設定（MVP では LLM 呼び出しのみ）
- エージェント間の動的な協調ロジック（MVP では固定のパイプライン構造）

### MVP スコープ外（明示的に含めないもの）

#### 機能スコープ外

- **UC #2, #3 の実装** — v2 以降で段階追加
- **AI によるチーム構成提案** — v2 で追加予定。MVP ではテンプレート起点のみ
- **テンプレートのユーザー作成・編集** — MVP ではシード テンプレートのみ提供
- **ユーザー認証・マルチユーザー** — MVP はシングルユーザー前提（ADR-0003 スコープ外と整合）
- **Web 検索・外部データソース連携** — MVP では LLM の知識ベース＋ユーザー提供の参考情報で調査。外部検索 API 連携は v2
- **エージェントの中断・再開・リトライ** — MVP では実行は一方向（開始→完了）

#### ADR-0003 から継承するスコープ外

- 音声・画像生成機能
- モバイル専用アプリ
- エンタープライズ機能（SSO、監査ログ、マルチテナント）
- 本番運用・課金・商用利用

### MVP 成功基準

ADR-0004 のペルソナ成功判定基準を MVP スコープに適用：

1. **定量**: プライマリペルソナの JTBD シナリオ（競合 3 社×4 観点の比較整理）を **30 分以内**に完走できる
2. **定量**: 手動比 **50% 以上**の時間削減が観測される（MVP 合格ライン。手動 3 時間→MVP 利用で 1.5 時間以内）
3. **定性（出力品質）**: 調査結果を基に提案資料のドラフトに着手できるレベルの構造・網羅性がある（ユーザー提供の参考情報を活用した場合）
4. **定性（体験）**: ドッグフーディングで PM / エンジニア自身が「手動より明確に楽」と判断できる

### v2 以降のロードマップ概観

MVP の検証結果を踏まえ、以下を段階的に追加する：

| フェーズ | 追加機能 |
| --- | --- |
| v2 | AI チーム構成提案、Web 検索連携、UC #2 追加、テンプレート作成・編集 |
| v3 | UC #3 追加、エージェント中断・再開、実行結果のフィードバックループ |

## Consequences

### ポジティブ

- Hero UC 1 本への集中により、エージェントチーム実行基盤を深く作り込める
- テンプレート起点の UX で出力品質が安定し、MVP 検証の信頼性が高まる
- リアルタイム進捗表示により「エージェントチームの協働」というコア価値が MVP 段階から体験できる
- 成功基準が JTBD の定量指標（時間削減）に直結しており、合否判定がシンプル

### ネガティブ / リスク

- UC 1 本のみのため、プラットフォームとしての汎用性は MVP 段階では検証できない（v2 で UC 追加時に検証）
- Web 検索 API なしのため、調査結果の鮮度は LLM の学習データとユーザー提供情報に依存する（v2 の Web 検索連携で対応）
- テンプレート起点のみでは「自分のチームを率いる」という北極星ビジョンとの距離が大きい（v2 の AI 提案機能で段階的に接近）

### 中立

- 技術スタックの選定（#10）、アーキテクチャ方針（#11）は本 ADR のスコープ外。MVP 機能一覧が入力となる
- MVP の Web 検索非対応は意図的なスコープカットであり、技術的制約ではない
