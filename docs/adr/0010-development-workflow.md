# 0010. 開発フローの選定

## Status

accepted

- 作成日: 2026-04-19
- 関連: ADR-0007（AI駆動開発アーキテクチャ）, Issue #35

## Context

ADR-0007 で AI駆動開発の枠組み（エージェント構成、チェックポイント方式、品質保証3層）が決まり、Issue #12 で開発環境（Bun workspace, TypeScript, Biome, Turborepo, CI）が整備された。

プロダクト実装に入る前に、1 Issue 内の実装ワークフロー（何をどの順で行うか）を確定する必要がある。ADR-0007 の品質保証第2層（テスト駆動）の具体化でもある。

## Considered Alternatives

主要な開発手法を評価した。AI駆動開発（ADR-0007）との相性も考慮している。

| 手法 | 概要 | 判定 | 根拠 |
| --- | --- | --- | --- |
| 型駆動（Type-Driven） | 型定義を設計の主要手段とする | **採用（主軸）** | TypeScript strict + packages/shared 型共有が既に設計方針。AI 協働でも型が仕様書として機能する |
| TDD（軽量版） | テスト先行で実装する | **採用** | Service 層のロジック検証に効果的。ADR-0007 の品質保証第2層を具体化する |
| DDD | ドメインモデル中心に設計 | 見送り | エンティティ 3 つの小規模ドメイン。戦術パターン（集約、値オブジェクト等）は過剰。AI駆動との相性は悪くないが、規模に見合わない |
| BDD | Given-When-Then 形式の仕様記述 | 見送り（エッセンスは取り入れ） | 下記で詳述 |
| API駆動（OpenAPI） | OpenAPI 仕様を先に書く | 見送り | AI は OpenAPI YAML の生成・解釈が得意だが、型駆動で同等の恩恵が得られる。YAML 管理コストが型定義を上回る |
| 仕様駆動 | 仕様を先に定義して実装 | 型駆動に統合 | TypeScript 型定義を「仕様」として扱うことで低コスト実現 |

### BDD と AI駆動の相性について

Issue での却下理由は「Gherkin のフォーマル記法は 1人+AI チームでオーバーヘッド」だが、AI が書き手になる場合、フォーマルな構造はむしろ強みに転じる。Given-When-Then の明確な入出力定義は AI にとって解釈しやすく、テスト生成との親和性も高い。

一方、現状の軽量 TDD でも Service 層のテストで振る舞いは検証できる。BDD フレームワーク（Cucumber 等）の正式導入は過剰だが、テスト記述に BDD 的なスタイル（describe/it で振る舞いを自然言語的に記述）を取り入れることで、AI駆動の利点を活かしつつオーバーヘッドを抑える。

## Decision

**型駆動 + 軽量 TDD** を採用する。AI駆動開発の枠組み（ADR-0007）の中で、実装フェーズの具体的な進め方を定める。

具体的な実装ワークフローとテスト戦略は [開発ワークフローガイド](../guides/development-workflow.md) に記載する。

実装フェーズを通じてワークフローの改善が必要になった場合は、本 ADR を更新する。

## Consequences

- 型定義が仕様書として機能し、AI との協働で認識齟齬が減る
- 軽量 TDD により Service 層の品質が自動検証され、ADR-0007 の品質保証第2層が具体化される
- BDD 的な記述スタイルの採用により、AI がテストの意図を理解しやすくなる。フレームワーク導入のオーバーヘッドはない
- E2E テストを見送るため、画面間の結合不具合は手動確認に依存する（MVP の 3 画面なら許容範囲）
- 実装経験を経てワークフローの調整が発生する可能性がある
