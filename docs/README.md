# ドキュメント

プロジェクトのドキュメントを管理するディレクトリです。

## ディレクトリ構成

| ディレクトリ | 内容 |
| --- | --- |
| [product](./product/README.md) | プロダクト要件（ユーザーストーリー・ユースケース） |
| [design](./design/README.md) | 設計ドキュメント |
| [adr](./adr/README.md) | Architecture Decision Records（意思決定記録） |
| [guides](./guides/README.md) | 運用ガイド（ブランチ戦略、GitHub 運用等） |

## 位置付け

- **product/**: ユーザーが何を達成したいか（What / Why）
- **design/**: それをどう実現するか（How）
- **adr/**: なぜその判断に至ったか（Decision / Rationale）
- **guides/**: チームとしてどう運用するか（Operations）

## 配置ポリシー

[ADR-0012](./adr/0012-doc-placement-policy.md) で定めた 2 つの原則に従う。

### 原則 1: 読み手テスト

| | product/ | design/ |
| --- | --- | --- |
| 読み手 | PO・PM・ビジネス | エンジニア・アーキテクト |
| 含む | 意図・振る舞い・UX・ビジネスルール | データ契約・型・アーキテクチャ・パラメータ |
| 含まない | JSON Schema・enum リテラル・型定義・API 仕様 | ユーザーストーリー・画面フロー・プロンプト本文 |

### 原則 2: コードが SSoT

コードで表現できることはコードを唯一のソースとする。

| フェーズ | SSoT | ドキュメントの役割 |
| --- | --- | --- |
| 設計段階（コード未存在） | ドキュメント（暫定 SSoT） | AI・人間への実装入力仕様 |
| 実装後 | コード（型・スキーマ・テスト） | コードへの参照 + コードで表現できない意図・背景 |

### 迷ったときの判断基準

1. 非技術者が読む必要があるか → product/
2. エンジニアが実装に使うか → design/
3. データ型を定義しているか（JSON Schema, enum, 型制約）→ 常に design/（実装後はコードが SSoT）
