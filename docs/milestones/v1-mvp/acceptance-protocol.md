# MVP 受入確認プロトコル

- **対象マイルストーン**: v1 - MVP Implementation
- **検証対象**: ADR-0005 成功基準 1（30 分以内完走）/ 成功基準 2（手動比 50% 削減）
- **根拠 Issue**: #198 / 子 Issue #200–203

---

## 1. 固定シナリオ

再現性を確保するため、手動・MVP 両計測で同一シナリオを使用する。

### 対象企業（3 社）

| # | 企業名 | 分類 |
| - | --- | --- |
| 1 | Dify | AI アプリ開発プラットフォーム |
| 2 | n8n | ワークフロー自動化（セルフホスト対応） |
| 3 | Zapier | ワークフロー自動化（SaaS） |

### 調査観点（4 種）

| # | 観点 | 調査内容の例 |
| - | --- | --- |
| 1 | 戦略 | 事業方向性・ミッション・最近の方針変更 |
| 2 | 製品 | 主要機能・UX・直近のリリース・差別化 |
| 3 | 投資 | 資金調達状況・ラウンド・財務健全性 |
| 4 | パートナーシップ | 提携・連携サービス・エコシステム拡大 |

### 参考情報

- 各社の最新プレスリリース・ブログ記事（直近 3 ヶ月分）からテキストを抜粋して参考情報欄に入力する
- URL ではなくテキストを貼り付ける（MVP 仕様に準拠。Web 取得は行わない）
- 両計測（手動 / MVP）で同一のテキストを使用する
- 参考情報の目安: 各社 300〜500 字程度

## 2. 手動ベースライン計測手順（C2 #201）

同一シナリオを LLM 単発チャットで実施する。

### 実施条件（手動）

- アプリ（Agent Team Studio）は使用しない
- 実施者: C3 と同一の 2 名（PM / エンジニア）が各自独立して実施
- 使用 LLM: Claude.ai（モデルは実施前に 2 名で合意の上で固定する）
- LLM チャット 1 セッションのみ使用（追加セッション不可）
- 参考情報テキストはプロンプトに含めてよい

### 計測手順（手動）

1. タイマーを開始する
2. LLM チャットに以下の内容を入力して送信する：
   - 調査対象（Dify / n8n / Zapier）
   - 調査観点（戦略 / 製品 / 投資 / パートナーシップ）
   - 参考情報テキスト
   - 出力形式の指示（観点×企業のマトリクス形式）
3. LLM の応答を確認し、不足があれば追加プロンプトで補完する
4. 結果が「提案資料のドラフトに着手できるレベル」に達したと判断した時点でタイマーを止める（目安：観点×企業のマトリクス全セルに内容が記載されていること）
5. 計測ログに記録する

### 記録項目（手動）

- 所要時間（分：秒）
- 追加プロンプト回数（手戻り回数）
- 出力の主観的網羅性（5 段階）
- 体感メモ（途中で感じた不安・迷い等）

## 3. MVP 実行計測手順（C3 #202）

同一シナリオを Agent Team Studio（MVP）で実施する。

### 実施条件（MVP）

- C2（手動ベースライン計測）が完了していること
- PR #190 (US-5) がマージ済みで全機能が利用可能な状態
- 2 名（PM / エンジニア）が同一アプリバージョンで独立して各自実施
- 各自が計測ログを記録した後にもう一方の結果を確認する（先行者の結果を見てから実施しない）

### 計測手順（MVP）

1. タイマーを開始する（アプリ起動前）
2. テンプレート選択画面で「競合調査テンプレート」を選択する
3. 入力フォームに以下を入力する：
   - 競合企業名: Dify / n8n / Zapier
   - 調査観点: 戦略 / 製品 / 投資 / パートナーシップ
   - 参考情報: 固定シナリオと同一テキスト
4. 実行を開始する
5. 結果画面のマトリクス全セルに内容が表示されていることを確認した時点でタイマーを止める。エラー・欠損セルがある場合は完走不能として記録する
6. 計測ログに記録する

### 記録項目（MVP）

- 所要時間（分：秒）
- 体感メモ（待ち時間中の不安・離脱衝動・操作迷い）
- エラー発生の有無と内容

## 4. 合否判定基準

### 成功基準 1: 30 分以内完走

| 判定 | 条件 |
| --- | --- |
| PASS | n=2 の平均が 30 分 00 秒以内 |
| CONDITIONAL PASS | n=2 の平均が 30 分 01 秒〜45 分。改善余地ありと記録し v2 で対応 |
| FAIL | n=2 の平均が 45 分超 or いずれか 1 名が完走できない |

### 成功基準 2: 手動比 50% 削減

削減率 = (手動時間 − MVP 時間) / 手動時間 × 100

| 判定 | 条件 |
| --- | --- |
| PASS | 削減率 50% 以上 |
| CONDITIONAL PASS | 削減率 30% 以上 50% 未満。原因切り分けと v2 改善方針を記録 |
| FAIL | 削減率 30% 未満（50% 目標に対し、v2 改善でリカバリできない水準と判断するライン） |

### 定性基準（ADR-0005 成功基準 3・4）のスコープ

成功基準 3（出力品質）・4（体感）は本プロトコルの定量計測スコープ外とする。C3 の体感メモと出力結果をもとに、C4 完了時に PM / エンジニアの合議で別途判定する。

### FAIL / CONDITIONAL 時の原因切り分け軸

- **モデル**: LLM の応答品質・速度が手動と比較して劣っているか
- **プロンプト**: エージェントのプロンプト設計に問題があるか
- **並列度**: 並列実行が時間短縮に寄与しているか（直列と比較）
- **UX**: 入力手順・待機体験・結果表示に改善余地があるか
