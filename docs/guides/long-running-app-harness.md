# 長時間稼働アプリ向け 3 エージェントハーネスの調査と適用設計

Issue [#270](https://github.com/kuairen-227/agent-team-studio/issues/270)（親トラッカ [#269](https://github.com/kuairen-227/agent-team-studio/issues/269)）の Spike 成果。Anthropic が示した長時間稼働アプリ開発向けの 3 エージェントハーネス（Planner / Generator / Evaluator）を精読し、本リポジトリの既存資産への適用マップ・PoC 最小スコープ・リスクと緩和策を整理する。**実装はスコープ外**で、採否と段階導入方針は [ADR-0038](../adr/0038-autonomous-agent-loop-adoption.md)（accepted）で扱う。

全体ハーネスにおける本ループの位置づけは [ai-driven-development.md](./ai-driven-development.md)（「今後の計画」節・施策インベントリ）を参照。

## 1. 記事の要点

出典: Anthropic, [Harness design for long-running application development](https://www.anthropic.com/engineering/harness-design-long-running-apps)（2026 年）。GAN（敵対的生成ネットワーク）に着想を得て、生成と評価を別エージェントに分離した構成。数時間規模の自律コーディングでフルスタックアプリを生成した実験の知見をまとめている。

### 3 エージェントの役割

| エージェント | 役割 |
| --- | --- |
| **Planner** | 1〜4 文の短いプロンプトを、フルのプロダクト仕様へ展開する。「スコープは野心的に」「詳細実装よりプロダクト文脈と高レベル技術設計に集中」と指示される。出力はコードではなく仕様。AI 機能の組み込み余地も仕様に織り込む |
| **Generator** | 仕様をスプリント単位で反復実装する。スプリント完了時に Evaluator へ引き渡す前に自己評価を行う。git でバージョン管理する |
| **Evaluator** | Playwright MCP で稼働中アプリをユーザーのように操作し、UI / API / DB 状態を検証する。スクリーンショットを撮り精査してから評価を下す。プロダクト深度・機能性・ビジュアル・コード品質などの基準で採点し、**いずれか 1 つでも合格しきい値（hard threshold）を下回ればそのスプリントは失敗** |

### スプリント契約（sprint contract）

各スプリント開始前に、Generator と Evaluator が「このチャンクの **done** とは何か」をコードを書く前に合意する。Generator が実装の具体と検証方法を提案し、Evaluator が要件との整合をレビューし、両者が合意するまで往復する。高レベルな仕様と「テスト可能な実装」の橋渡しを担う。契約は粒度が細かく、レトロゲームメーカーの例では **Sprint 3 だけで 27 個** の判定基準があった（例: 「矩形塗りつぶしツールがドラッグで領域を塗る」→ FAIL: ドラッグ始点/終点にしかタイルを置かない）。

### 計画 → 実装 → 検証 → 修復ループ

Planner が仕様を作る → Generator と Evaluator がスプリント契約を交渉 → Generator が実装 → Evaluator が契約基準で検証 → 詳細フィードバック → Generator が反復。エージェント間通信は **ファイル経由**（一方が書き、他方が読む）。

### 検証のしかた

- **ツール**: Playwright MCP で Evaluator が自らページを操作・スクリーンショット・精査してから評価を出す。
- **基準の較正**: デザイン系 4 基準（品質・独創性・作り込み・機能性）を few-shot の採点例で較正した。主観的判断（「良いデザインか？」）を具体的で採点可能な基準へ落とすことが鍵。
- **懐疑性**: 初期の Evaluator は「正当な問題を見つけても、大したことではないと自分を納得させてしまう」傾向があった。評価結果と望ましい結果の乖離をレビューしてプロンプトを反復改善することで矯正した。

### オーケストレーションとコンテキストの受け渡し

ループは **Claude Agent SDK** 上に構築され、SDK の自動コンパクションがコンテキスト増大を処理する。状態は構造化ファイルでハンドオフする。

ここで **コンテキストリセット**（コンテキストを完全にクリアし、構造化ハンドオフで状態を引き継いだ新エージェントを起動）と **コンパクション**（同じエージェントが履歴を要約して継続）は別物。Sonnet 4.5 は「コンテキスト不安（context anxiety）」が強く、コンパクションだけでは不十分でリセットが必須だった。**Opus 4.5 ではこの挙動がほぼ解消され、リセットを撤去して単一連続セッション + 自動コンパクションで全ビルドを走らせた**。

### ハーネスの進化（V1 → V2）とモデル世代依存性

記事の最重要教訓: **ハーネスの各部品は「モデルが単独でできないこと」の仮定であり、モデルが良くなると陳腐化する**。指針は「最も単純な解から始め、必要なときだけ複雑さを足す」（Building Effective Agents）。新モデルが出たら、効かなくなった足場を剥がし、新たに可能になった能力を足してハーネスを再点検する。

| 版 | モデル | 構成 | 例 / コスト |
| --- | --- | --- | --- |
| **V1** | Opus 4.5 | 3 エージェント + **スプリント**（契約・per-sprint 採点） | レトロゲームメーカー。単独 20 分/$9 に対しフルハーネス **6 時間/$200**（20 倍超） |
| **V2** | Opus 4.6 | **スプリント撤廃**。Planner / Evaluator は維持、Evaluator は**最後の単一パス**採点へ | DAW。**約 3 時間 50 分/$124.70**（Build が 2 時間超を一貫実行） |

- V2 で Planner を外すと Generator はスコープを過小に取る（仕様化せず実装を始める）ため、**Planner と Evaluator は引き続き価値がある**。
- **Evaluator は固定の yes/no 判断ではない**。タスクがモデルの単独能力を超える領域にあるときコストに見合う。モデルが強くなるほどその境界は外へ動き、境界内のタスクでは Evaluator は過剰オーバーヘッドになる。
- 記事中に **明示的な中断機構・無限ループ抑止・人間チェックポイント** の記述はない（本リポジトリ適用時に補う論点 → §4）。

## 2. 適用方針：best-practice 優先で purpose-built

記事のハーネスは既存ツールの寄せ集めではなく、**専用エージェント（dedicated personas）を Claude Agent SDK 上に新規構築**したもの。本リポジトリでも **best practice を優先し、3 エージェントは purpose-built で設計する**。既存のロールベースエージェント（`qa` 等、[ADR-0011](../adr/0011-role-based-agent-architecture.md)）や人手前提スキル（`implement-feature` / `review`）は **再利用を強制する制約ではなく、参照・流用してよい素材** として扱う。とくに `qa` は human-in-the-loop のレビュー視点で、自律 Evaluator（較正済みルーブリック + hard threshold + Playwright 自走）とは意図が異なるため、**役割を流用せず別物として作る**。

### purpose-built で作るもの（記事準拠の中核）

| エージェント | best-practice 設計 |
| --- | --- |
| **Planner** | 1〜4 文 → 仕様の専用プロンプト。野心的スコープ、product 文脈と高レベル設計に集中（詳細実装は下流に委ね、誤りの連鎖を避ける）。AI 機能の織り込み |
| **Generator** | 仕様に対し自律実装する専用エージェント。自己評価してから Evaluator へ。git 管理。型駆動 / 軽量 TDD の規律を**参照**してよいが、`implement-feature`（人手手順）に縛られない |
| **Evaluator** | 較正済み採点ルーブリック（few-shot + 詳細スコア内訳）と **hard threshold**、Playwright で自走検証する専用エージェント。懐疑性をプロンプトで作り込む（素の Claude は QA が甘い）。`qa` を流用せず新規 |

### 流用してよい素材（ツール・基盤・規律）

これらは「ベストプラクティスそのもの」なので積極的に使う。アーキテクチャを縛るためではなく、使える部品として:

| 素材 | 流用理由 |
| --- | --- |
| **Playwright MCP**（[ADR-0024](../adr/0024-playwright-mcp-for-ai-verification.md)） | 記事の Evaluator と同じ検証手段。そのまま中核ツール |
| **Claude Agent SDK** | 記事の実証オーケストレーション基盤（自動コンパクション含む）。§5 参照 |
| **ファイルハンドオフ基盤**（`docs/` / Issue / PR） | 構造化ハンドオフの実体。エージェント間通信に流用 |
| **安全網**（egress firewall + `permissions.deny` + DevContainer、[ADR-0037](../adr/0037-ai-execution-sandbox-policy.md)） | 無人実行の前提。そのまま前段に置く |
| **中断機構**（AbortController / `AbortSignal.any`、[ws-llm-spike.md](../design/technotes/ws-llm-spike.md)） | 実証済みパターンを流用 |

要点: 記事の最大の知見（生成と評価を**別エージェントに分離**）を最優先し、Evaluator は `qa` の延長ではなく**専用に作り込む**。既存資産は再利用を前提にせず、ベストプラクティスに合うものだけ部品として使う。

## 3. PoC 最小スコープ（段階案）

全 3 エージェント（Planner / Generator / Evaluator）導入を目標に据えつつ、リスクとコストの低い順に段階的に積む。記事の指針に従い **最も単純な構成から始め、必要なときだけ足場を足す**。とくに **スプリント機構は必須ではなくモデル能力依存の足場**（記事 V2 では撤廃）なので、まず足場なしで始め、対象タスクがモデルの単独能力を超えて初めて導入を検討する。各 Phase は独立に価値を出し、次へ進む判断材料を残す。

| Phase | スコープ | 狙い / 着手しやすさ |
| --- | --- | --- |
| **Phase 0**（現状） | 人手駆動の疑似ループ（`process-issue` → `implement-feature` → `review`）。ベースライン | ベースライン（実装不要） |
| **Phase 1: Evaluator** | **専用 Evaluator エージェントを新規構築**（較正ルーブリック + hard threshold + Playwright 自走）。まずは**最後の単一パス**採点（記事 V2 相当）。既存の人手レビュー（`review` / `qa`）はフォールバックとして併存 | 記事の最大の知見（自己評価の分離）を最小コストで検証。`qa` の延長ではなく専用に作る。Evaluator 単体コストはフルの約 8% 相当（V2 実測、§4）で、着手 Issue のコスト見積もりの出発点になる |
| **Phase 2: Generator** | **専用 Generator エージェント**で Planner 仕様を自律実装（型駆動 / 軽量 TDD を規律として参照）。反復上限・コスト上限を機械的に課す。**スプリント契約 + per-sprint 採点（記事 V1 相当）は、タスクがモデル単独能力を超える場合にのみ足す**。運用設計は §8 | 自律実装の暴走/コスト挙動をここで測る |
| **Phase 3: Planner + 統合** | **専用 Planner**（プロンプト/Issue → 仕様）+ 3 者ループ統括。**全採用の到達点** | フル自律。オーケストレーションは §5 の再評価で Claude Agent SDK / 自前を確定 |

各 Phase の Go/No-Go は、コスト実測・合格率・人手介入頻度を見て判断する。Evaluator は固定で挟むのではなく、タスクがモデル単独能力を超えるときに投入する（§1 の教訓）。エージェントは既存スキルの流用を前提とせず purpose-built で作り、Playwright MCP・Agent SDK・安全網など§2 の素材のみ流用する。

## 4. リスクと緩和策

| リスク | 緩和策 |
| --- | --- |
| **コスト**（フルで $124〜200/run） | トークン/時間バジェットの上限設定。Phase で段階課金を観測。高コストな Opus フル稼働は対象タスクの規模で正当化できる範囲に限定（記事「小タスクでは Evaluator は過剰」） |
| **モデル階層化**（コスト最適化） | コストはほぼ Generator に偏る（V2 実測で約 91%。Planner 約 0.4% / Evaluator 約 8%）。**Generator は能力境界内の定型タスクで Sonnet 併用を検討**し、Planner / Evaluator は判断品質・連鎖影響を優先して強モデル（Opus）に据え置く。ただし Generator を弱めると Evaluator の反復が増え QA 往復で食い潰し得るため、合格率・総コスト（QA 込み）を Phase 2 で実測して採否を決める。ループはエージェント単位でモデル指定可能（model-agnostic）に設計する。プロダクト側のマルチベンダー方針（[ADR-0032](../adr/0032-llm-multi-vendor-strategy.md) / [ADR-0034](../adr/0034-llm-client-ai-sdk.md)）とは別物のハーネス側選択 |
| **暴走・無限ループ** | 合格しきい値の機械化 + **最大スプリント数 / 最大反復回数の上限**。Evaluator の懐疑性チューニング（記事の「自分で問題を見逃す」教訓）。しきい値を下回り続ける場合は人手へエスカレーション |
| **中断（AbortSignal）** | 実行中断は AbortController / `AbortSignal.any`（多階層 signal、[ws-llm-spike.md](../design/technotes/ws-llm-spike.md) で実証済みパターン）。tool レベルの遮断は `permissions.deny` |
| **人間チェックポイント** | 記事に明示機構がない分を本リポジトリの **PR / Issue ゲート** で補う。スプリント契約合意・PR 作成・ADR 要否の各点で human-in-the-loop を置く |
| **データ持ち出し（exfiltration）** | [ADR-0037](../adr/0037-ai-execution-sandbox-policy.md) の egress allowlist firewall（default-deny）。無人・長時間ほど価値が上がる。ADR-0037 が残した **fail-close（適用失敗時のコンテナ停止）/ 起動ヘルスチェックの要否**は、本ループ無人実行の設計時に再評価する |
| **状態の継承漏れ** | 構造化ファイルハンドオフ（記事準拠）。Opus 4.5 はコンテキストリセット不要のため、リセット跨ぎ設計の負荷は下がる |

## 7. 専用 Evaluator の運用設計

Issue [#289](https://github.com/kuairen-227/agent-team-studio/issues/289) で構築する専用 Evaluator（[.claude/agents/evaluator.md](../../.claude/agents/evaluator.md)）の運用方針を定める。設計は Anthropic 公式の長時間稼働エージェント実装（[cwc-long-running-agents](https://github.com/anthropics/cwc-long-running-agents) の `evaluator.md`：懐疑的・read-only・PASS/NEEDS_WORK・Default-FAIL・証拠ゲート）を骨格に、記事の上乗せ要素（較正済みルーブリック + 基準ごとの hard threshold + Playwright 自走）を加えたもの。本節は SSoT（段階導入ロードマップ上の位置づけは §3 を参照）。**しきい値・定量基準の数値は記事の実測を出発点にした初期ドラフトであり、運用の実測で調整する**。なお Evaluator の DB 参照は SELECT のみとする規約だが、ローカル DB 前提のため実行レイヤでの強制（read-only 接続等）は設けない。本番 DB を扱う段階で読み取り専用接続（`DATABASE_READONLY_URL` 等）を再検討する。

### 7.1 投入対象タスクの範囲（規模・種別）

記事の対象は数時間規模の新規生成で、本リポジトリの日常的 Issue 実装とは規模が異なる。Evaluator は固定で挟むものではなく、**タスクがモデルの単独能力の境界上にあるときに投入する**（§1 の教訓）。

| 区分 | 対象 |
| --- | --- |
| **投入する**（境界上） | UI + API + DB を横断する `enhancement`。視覚 / 操作の動作確認が要る機能。複数ファイル・複数レイヤにまたがる実装。受入条件が 3 件以上で、自走検証で pass/fail を判定できるもの |
| **投入しない**（過剰オーバーヘッド） | 単一関数の軽微な `bug` 修正。docs / ADR / 型のみの変更。UI / API / DB に影響しない `chore`（設定・ツーリング）。人手の単体・統合テストで十分に担保できる変更 |

判定が曖昧なタスクは、まず投入せず人手レビューで進め、見逃しが続いた領域を投入対象に追加する。

### 7.2 較正済みルーブリックと hard threshold

採点基準・スコア尺度・hard threshold・few-shot 較正例の本体は [.claude/agents/evaluator.md](../../.claude/agents/evaluator.md) を SSoT とする（本節はサマリであり、evaluator.md と矛盾する場合は **evaluator.md を正**とする）。要点:

- 5 基準（受入条件の充足 / 機能正当性 / コード品質 / リグレッション・安全性 / UX・ビジュアル）を 1〜5 で採点。
- **基準ごとに hard threshold を置き、1 つでも下回れば総合 NEEDS_WORK**（受入条件の充足は 5、他は 4。UX は UI 時のみ）。
- 素の Claude の甘さ（問題を見つけても自分を納得させて承認する）を few-shot の較正例で矯正する。評価結果と望ましい結果の乖離が出たら、その事例を較正例に追記してプロンプトを反復改善する。

### 7.3 Evaluator と人手レビューの優先順序・エスカレーション基準

現状（Evaluator のみ自動化し、生成は人手）は専用 Evaluator と人手レビュー（[`review`](../../.claude/skills/review/SKILL.md) / [`qa`](../../.claude/agents/qa.md)）が併存する。**人手レビューが最終権限**（human-in-the-loop。ADR-0038 ガードレール）。

優先順序:

1. Evaluator が**一次ゲート**として機械判定（PASS / NEEDS_WORK）。
2. NEEDS_WORK は builder に差し戻し（人手レビューに乗せない）。
3. PASS のものを**人手が最終確認**（PR レビュー）。Evaluator はフォールバックではなく前段、人手が最終確認という関係。

エスカレーション（判定不一致時）:

| ケース | 処理 | 記録先 |
| --- | --- | --- |
| **Evaluator PASS → 人手が否定**（偽陰性 / 見逃し） | PR を差し戻し。当該事例を 7.2 の較正例に追記してルーブリック / プロンプトを更新 | 7.5 の計測ログ + [ADR-0007](../adr/0007-ai-driven-dev-architecture.md) 第 3 層 Phase 3（実行メトリクス蓄積によるプロンプト改善）の記録 |
| **Evaluator NEEDS_WORK → 人手が「実は合格」**（偽陽性 / 過剰懐疑） | 人手判断を優先しつつ、過剰懐疑の事例として較正例に追記 | 同上 |
| **しきい値を下回り続ける / 収束しない** | 人手へエスカレーション（§4 の暴走・無限ループ緩和）。現状（Evaluator 単体）の「収束しない」は同一成果物への NEEDS_WORK が 3 回連続した状態を目安とする | 同上 |

### 7.4 次段階（Generator 自律化）への Go/No-Go 定量基準

ADR-0038 Decision §5 が必須とする定量基準。**初期ドラフト（記事実測が出発点）**。本運用の実測がこれを満たして初めて次段階（Generator の自律化）に着手する。

| 指標 | Go 条件（初期ドラフト） | 根拠 |
| --- | --- | --- |
| **コスト上限** | 1 評価あたり ≤ 約 $10 | 記事 V2 で Evaluator はフル run の約 8%（フル $124.70 → 約 $10）。本リポジトリの Issue は数時間規模より小さいため通常はこれを大きく下回るはず。超過が常態化したら投入対象が過大 → 見直し |
| **判定一致率** | Evaluator 判定と人手最終判定の一致率 ≥ 80%、かつ偽陰性率（Evaluator PASS を人手が否定）≤ 10% | 自己評価分離の価値を担保する最低ライン。偽陰性は最も危険なため別途上限を置く |
| **人手介入頻度** | Evaluator フィードバックだけで次アクションが定まった割合 ≥ 80%（指摘を解釈し直す手戻り ≤ 20%） | フィードバックの具体性 = Generator 自律化の前提。現状は Generator が人手のため「フィードバックの具体性」で代理測定 |

**蓄積前の運用**: 評価件数が少ないうちは一致率・偽陰性率が統計的に意味をなさないため、**評価件数 n ≥ 10 に達するまでは上記の定量基準を Go 判断に用いず、人手の定性判断を継続する**（測定可能になる前に Go を出さない）。

いずれかを外したら方針を見直す（ADR-0038 再検討契機）。

### 7.5 計測（コスト / 合格率 / 人手介入頻度）と記録様式

現状は手動運用のため、1 評価ごとに以下を**対象 PR / Issue のコメント**に残し、Go/No-Go 判断時に集計する。

| フィールド | 内容 |
| --- | --- |
| 日付 / 対象 | 評価日・対象 Issue / PR |
| スコア内訳 | 5 基準のスコアと総合判定（PASS / NEEDS_WORK） |
| 人手最終判定 | PASS / NEEDS_WORK と Evaluator 判定との一致 / 不一致 |
| コスト | 推定トークン / 所要時間（→ 7.4 コスト上限） |
| 人手介入回数 | ステアリング・中断・手動修正の回数（→ 7.4 介入頻度） |
| フィードバック有効性 | 指摘だけで次アクションが定まったか（y/n。→ 7.4 介入頻度） |

集計値（一致率・偽陰性率・平均コスト・介入頻度）を 7.4 の Go 条件と突き合わせて次段階（Generator 自律化）の着手可否を判断する。

## 8. 専用 Generator の運用設計

Issue [#290](https://github.com/kuairen-227/agent-team-studio/issues/290) で構築する専用 Generator（[.claude/agents/generator.md](../../.claude/agents/generator.md)）の運用方針を定める。設計は §2 の purpose-built 方針に従い、仕様（受入条件）を型駆動 / 軽量 TDD の規律で自律実装し、自己評価のうえ [§7](#7-専用-evaluator-の運用設計) の Evaluator へファイルハンドオフする builder として作る。人手手順スキル [`implement-feature`](../../.claude/skills/implement-feature/SKILL.md) とは別系統で、その規律（型駆動・軽量 TDD・層順序）は参照しつつ、人手チェックポイントには縛られない。本節は SSoT（段階導入ロードマップ上の位置づけは §3 を参照）。コストはほぼ Generator に偏る（V2 実測で約 91%、§4）ため、暴走・コスト挙動・モデル選択をここで実測する。**しきい値・定量基準の数値は記事の実測を出発点にした初期ドラフトであり、運用の実測で調整する**。

**実行モデル（逐次）**: Phase 2 は Generator と Evaluator を**同時に走らせない**。Generator が実装・自走検証（`bun run dev` での Playwright 確認を含む）を完了し dev サーバを停止してから、Evaluator を起動する。これによりポート競合・アプリ状態の相互干渉を避ける（[generator.md](../../.claude/agents/generator.md) 手順 6 の「検証後は dev サーバを停止する」と対）。

### 8.1 投入対象タスクの範囲

Generator を投入する範囲は §7.1 の Evaluator 投入範囲と整合させる（Evaluator が機械採点できるタスク = Generator が自律実装に挑む価値があるタスク）。

| 区分 | 対象 |
| --- | --- |
| **投入する**（境界上） | 受入条件が機械的に実装/検証へ落とせる `enhancement`。UI + API + DB を横断し、Playwright 自走・DB SELECT・テストで pass/fail を判定できるもの |
| **投入しない**（自律化の利得が薄い） | 仕様が曖昧で実装方針が一意に定まらないもの。docs / ADR / 設計判断そのもの。人手の単体・統合テストで十分担保できる軽微な `bug` / `chore` |

仕様が曖昧なタスクは、Generator が推測実装に走るとスコープがぶれるため**投入せず人手で仕様を固めてから**回す（Planner 不在の Phase 2 では仕様の明確化が人手の責務）。

### 8.2 反復・コスト・時間の機械的上限

ADR-0038 ガードレール「反復/コスト上限」の具体化。暴走・無限ループ抑止のため機械的に課す。**初期ドラフト**。

| 上限 | 値（初期ドラフト） | 根拠 / 措置 |
| --- | --- | --- |
| **最大反復回数** | 同一仕様の Generator↔Evaluator 往復 ≤ 5 回 | 5 回で収束しなければ仕様/タスクがモデル単独能力を超える兆候。人手へエスカレーション |
| **収束失敗の打ち切り** | 同一成果物への NEEDS_WORK が **3 回連続** で人手へ | §7.3 の Evaluator 側「収束しない」目安と対称に揃える |
| **トークン/時間バジェット** | 初期ドラフト: 1 自律実装 run あたり **≤ 約 $113 / 約 3 時間 45 分**（kill-switch の天井。n≥10 で実測調整） | 記事 V2 のフル run 約 3 時間 50 分 / $124.70 のうち Generator が約 91%（≈ $113）。これは暴走を止める**絶対上限**で、§8.6 の Go 目標（平均 ≤ 約 $50/run）とは別物。本リポジトリの Issue はより小規模のため通常は大きく下回る想定で、超過が常態化したら投入対象が過大 → 見直し |
| **中断機構** | AbortController / `AbortSignal.any`（多階層 signal）、tool 遮断は `permissions.deny` | [ws-llm-spike.md](../design/technotes/ws-llm-spike.md) の実証パターンを流用 |

上限到達時、Generator は自力続行せず中断・報告する責務を負う（[generator.md](../../.claude/agents/generator.md) ガードレール）。

### 8.3 スプリント機構の要否（判断）

**判断: Phase 2 ではスプリント契約 + per-sprint 採点を採用しない（単一パス + 反復で始める）。**

- 記事の最重要教訓「最も単純な解から始め、必要なときだけ複雑さを足す」に従う。スプリント機構はモデル能力依存の足場で、記事 V2（Opus 4.6）では**撤廃**された（§1）。
- Phase 2 の出発点は「仕様 → 自律実装 → 自己評価 → Evaluator 単一パス採点 → 反復修復」。スプリント契約（実装前の done 合意の往復）は挟まない。
- **追加トリガ**: §8.2 の打ち切り（NEEDS_WORK 3 連続）が**同一種別の複数 Issue にまたがって起きた**場合に限り、その種別へスプリント契約の導入を検討する（単発の収束失敗では足さない）。導入は人手判断（チェックポイント）を経る。

### 8.4 fail-close / 起動ヘルスチェックの要否（ADR-0037 帰着）

[ADR-0037](../adr/0037-ai-execution-sandbox-policy.md) が「無人実行の設計時に再評価する」として #270 → 本 Phase へ残した宿題の帰着先（ADR-0038 Consequences が Phase 2 必須検討事項に指定）。

**判断:**

| 機構 | Phase 2 の判断 | 理由 |
| --- | --- | --- |
| **起動ヘルスチェック** | **採用**。ループ起動前に egress firewall（ADR-0037 の安全網）が有効であることを確認し、無効なら**起動しない（fail-fast）** | 低コストで安全網の前提を担保。無効な状態で自律実行を始めるとデータ持ち出しリスクが無防備になる |
| **fail-close（実行中の適用失敗時にコンテナ停止）** | **Phase 2 では自動機構として見送り**。起動時チェック + 中断機構（AbortController）+ human-in-the-loop（PR ゲート）で代替 | Phase 2 は人手起動・監督下の運用で、実行中に firewall が落ちる事象は監督で検知可能。自動コンテナ停止の複雑さは「必要時のみ足す」原則に照らし時期尚早 |

**再評価契機**: 完全無人化に近づく Phase 3（Planner + 統合）着手時に fail-close を再評価する。Phase 2 の運用で監督外の時間帯に実行する判断をした場合も再評価する。

### 8.5 ハーネス側モデル選択（Sonnet 併用可否）の実測判断

ADR-0038 Decision §5 / §4 が Phase 2 で実測判断するとした論点。コストはほぼ Generator に偏る（§4 のとおり）ため、Generator のモデル階層化が最大の最適化余地。**ハーネス側のモデル選択であり、プロダクト側マルチベンダー方針（[ADR-0032](../adr/0032-llm-multi-vendor-strategy.md) / [ADR-0034](../adr/0034-llm-client-ai-sdk.md) の `llm-client.ts` 公開境界）とは独立**。

**判断（実測プロトコル）:**

- **設計**: ループはエージェント単位でモデル指定可能（model-agnostic）に作る。Planner / Evaluator は判断品質・連鎖影響を優先し強モデル（Opus）に据え置く。
- **初期スタンス**: Generator も**まず Opus で計測を開始**する（Sonnet 併用を正当化する実測データが現時点でないため、計測の出発点を強モデルに置く）。
- **採否基準**: Generator を Sonnet に下げた場合の **合格率（最終 PASS 率）と総コスト（Evaluator 反復・QA 往復込み）** を Opus と比較する。Generator を弱めると Evaluator の反復が増え QA 往復でコストを食い潰し得るため、**単価ではなく総コストで採否を決める**。能力境界内の定型タスク（CRUD・定型 UI 等）で先行検討する。
- **蓄積前ロック**: §8.7 の評価件数 **n ≥ 10** に達するまで合格率・総コストの統計判断を行わず、初期スタンス（Opus）を継続する。

### 8.6 Phase 3（Planner + 統合）への Go/No-Go 定量基準

ADR-0038 Decision §5 が必須とする「次 Phase 着手前の定量基準」。**初期ドラフト（記事実測が出発点）**。本運用の実測がこれを満たして初めて Phase 3（専用 Planner + 3 者ループ統括）に着手する。

| 指標 | Go 条件（初期ドラフト） | 根拠 |
| --- | --- | --- |
| **最終 PASS 率** | 反復後に Evaluator PASS へ収束した割合 ≥ 80%、かつ平均反復回数 ≤ 3（§8.2 上限 5 の内側） | 自律実装が安定して収束することの担保。収束しない自律化は Phase 3 の前提を欠く |
| **自律完了率（人手介入頻度）** | 人手ステアリング・手動修正なしで完了した割合 ≥ 70%（手戻り ≤ 30%）。**分母 = 完了試行数（人手エスカレーションした run も含む全 run）** | 自律性の実効。Generator の自己修復がフィードバックだけで回るか。分母から人手エスカレーション分を除くと達成が容易になりすぎるため、全 run を分母とする |
| **コスト上限** | 1 自律実装 run あたり ≤ 約 $50（運用初期の暫定値、実測で確定） | Generator はコストの約 91%。記事 V2 フル $124.70 を上限の天井に置きつつ、本リポジトリの小規模 Issue では大きく下回る想定。超過が常態化したら投入対象が過大 → 見直し |

**蓄積前の運用**: 評価件数が統計的に意味を持つまで（**n ≥ 10**）は上記を Go 判断に用いず、人手の定性判断を継続する（§7.4 と同じロック）。いずれかを外したら方針を見直す（ADR-0038 再検討契機）。

### 8.7 計測と記録様式

現状は手動運用のため、1 自律実装 run ごとに以下を**対象 PR / Issue のコメント**に残し、Go/No-Go 判断時に集計する（§7.5 の Evaluator 側計測と対で蓄積する）。

**何を 1 件と数えるか**: 1 件 = 1 自律実装 run（1 仕様に対する Generator の 1 回の実装試行。反復は同一 run 内に含む）。§8.5（モデル比較）と §8.6（Go/No-Go）の n は**この同一カウンタを共有する**。§7.4 の Evaluator 判定数（n）とは別系統で独立に数える。§8.6 の自律完了率の分母は、人手エスカレーションした run も含む全 run（§8.6）。

| フィールド | 内容 |
| --- | --- |
| 日付 / 対象 | 実装日・対象 Issue / 仕様 |
| 反復回数 | Generator↔Evaluator 往復回数（→ §8.2 / §8.6） |
| 最終判定 | 収束先（PASS / 人手エスカレーション） |
| コスト | 推定トークン / 所要時間（→ §8.6 コスト上限） |
| モデル | Generator に用いたモデル（→ §8.5 実測） |
| 人手介入回数 | ステアリング・中断・手動修正の回数（→ §8.6 自律完了率） |

集計値（最終 PASS 率・平均反復回数・自律完了率・平均コスト・モデル別比較）を §8.6 の Go 条件・§8.5 の採否基準と突き合わせて Phase 3 着手可否を判断する。

## 9. 専用 Planner の運用設計と 3 者ループ統括

Issue [#291](https://github.com/kuairen-227/agent-team-studio/issues/291) で構築する専用 Planner（[.claude/agents/planner.md](../../.claude/agents/planner.md)）の運用方針と、Planner / Generator / Evaluator を束ねる 3 者ループの統括設計を定める。3 者の分離が揃い、フル自律ループが成立する**到達点**にあたる（段階導入ロードマップ上の位置づけは §3）。設計は §2 の purpose-built 方針に従い、Planner は短い入力（1〜4 文 / Issue）を**機械検証可能な仕様（受入条件）**へ展開する spec author として作る。ロールベース [`po`](../../.claude/agents/po.md)（human-in-the-loop のプロダクト判断）とは別系統で、その判断軸は参照しつつ、人手チェックポイントの外で実装手順に縛られない。本節は SSoT。**しきい値・定量基準の数値は記事の実測を出発点にした初期ドラフトであり、運用の実測で調整する**。

### 9.1 投入対象タスクの範囲

Planner は固定で挟むものではなく、**仕様化そのものが価値を生むときに投入する**。

| 区分 | 対象 |
| --- | --- |
| **投入する** | 1〜4 文の短いプロンプトや粒度の粗い Issue で、野心的スコープの新規 / 拡張。仕様化を経て §7.1 / §8.1 の投入対象（UI + API + DB 横断・受入条件 3 件以上・自走検証で pass/fail 可能）へ落とせるもの |
| **投入しない** | 既に受入条件が明確な Issue（Planner を介さず直接 Generator へ）。docs / ADR / 設計判断そのもの。仕様化の余地がない軽微な変更 |

### 9.2 Planner の出力（仕様）の様式

仕様は Generator（自律実装）と Evaluator（機械採点）の**契約**である。様式の本体は [.claude/agents/planner.md](../../.claude/agents/planner.md) を SSoT とする（本節はサマリ）。要点:

- **野心的スコープ・高レベルに留める**。product 文脈と高レベル技術設計に集中し、詳細実装は下流へ委ねる（誤った前提の連鎖を避ける）。
- 出力は **目的 / ユーザー価値・スコープ（含む / 含まない）・高レベル設計方針・AI 機能の織り込み・機械検証可能な受入条件・オープン issue / 人手判断ポイント**。
- 受入条件は Evaluator が証拠（snapshot / DB SELECT / テスト）で採点でき、Generator が実装単位に落とせる粒度にする（§9.1 の投入対象と整合）。

### 9.3 3 者ループ統括（計画 → 実装 → 検証 → 修復）

lead セッション（メインの Claude Code セッション）が 3 サブエージェントを**逐次**起動し、状態を**ファイル**で受け渡す（駆動主体・手段は §9.4 / [ADR-0043](../adr/0043-autonomous-loop-orchestration-mechanism.md)）。

| 段 | 駆動主体 | 入力（読む） | 出力（書く） | 人間チェックポイント |
| - | --- | --- | --- | --- |
| **計画** | Planner | 短いプロンプト / Issue・product 文脈 | 仕様ファイル（受入条件） | **仕様合意**（合意前に Generator を起動しない） |
| **実装** | Generator | 仕様ファイル | 作業ツリー + コミット + ハンドオフレポート（§8 様式） | ── |
| **検証** | Evaluator | 仕様・diff・稼働アプリ | 採点レポート（PASS / NEEDS_WORK・§7 様式） | ── |
| **修復** | Generator ↔ Evaluator | 採点レポート | 反復した実装 / 再採点 | 反復上限・収束失敗でエスカレーション（§8.2） |
| **完了** | 人手 | PASS した成果物 | PR | **PR レビュー**（人手が最終権限） |

実行は**逐次**で、Generator と Evaluator を同時に走らせない（§8 の逐次実行モデル：Generator が dev サーバを停止してから Evaluator を起動し、ポート競合・状態干渉を避ける）。エージェント間通信はファイル経由（一方が書き、他方が読む）。

### 9.4 オーケストレーション手段の確定（ADR-0038 論点 3）

**確定: 自前（Claude Code サブエージェント + ファイルハンドオフ）**。本体は [ADR-0043](../adr/0043-autonomous-loop-orchestration-mechanism.md) を SSoT とする。要点:

- 逐次・結果重視・依存の多い作業に対する公式の best fit はサブエージェント（main 統括・低コスト）。Agent Teams は並列・独立探索向けで実験的・高コストのため不適。
- 記事は Claude Agent SDK 採用だが orchestration 自体は straightforward で、SDK の load-bearing な価値（数時間単一セッションの自動コンパクション・durable execution）は本リポジトリの監督下・小規模運用では未到達。「最も単純な解から始め、必要なときだけ複雑さを足す」（§1）。
- **移行トリガ**（無人多時間運用の常態化 / コンテキスト破綻 / durable execution の要件化 / 自前保守コストの逆転）に到達したら Claude Agent SDK を再評価する（ADR-0043 Decision §3）。ループは model-agnostic 設計を維持し移行を阻害しない。

### 9.5 人間チェックポイントと安全網

ADR-0038 ガードレール「人間チェックポイント」「安全網」の Phase 3 具体化。

- **人間チェックポイント**: ①仕様合意（Planner 出力 ── 合意前にループを自走させない）、②PR 作成、③ADR 要否、④反復上限 / NEEDS_WORK 3 連続でのエスカレーション（§8.2）。**人手が最終権限**（human-in-the-loop）。
- **安全網**: egress firewall（[ADR-0037](../adr/0037-ai-execution-sandbox-policy.md) / [ADR-0041](../adr/0041-egress-firewall-nftables-ipv6.md)）の有効を前提とし、ループ起動前に**起動ヘルスチェック**を行う（§8.4）。
- **fail-close の Phase 3 再評価（§8.4 帰着）**: **自動 fail-close は Phase 3 でも引き続き見送る**。Phase 3 も自前オーケストレーションによる人手監督下運用（仕様合意・PR ゲート）で、実行中の firewall 失効は監督で検知可能なため。再評価は**完全無人化（ADR-0043 の Agent SDK 移行トリガ到達）と同期**して行う。

### 9.6 見送り候補（#225）の再評価

ADR-0038 Decision §4 / §5 が Phase 3 で再評価するとした候補の帰着。

| 候補 | Phase 3 での再評価 |
| --- | --- |
| **auto memory 活用**（B 群） | **引き続き見送り**。構造化ファイルハンドオフ（仕様ファイル / レポート）で 3 者間の状態継承は足りており、監督下ループで継承漏れは顕在化していない。セッション跨ぎの状態継承が要件化する（= ADR-0043 の無人化トリガ）段階で再判定 |
| **Managed Agents SDK 移行**（C 群） | **引き続き見送り**。オーケストレーション手段を [ADR-0043](../adr/0043-autonomous-loop-orchestration-mechanism.md) で自前に確定。Agent SDK 移行トリガ（無人多時間 / コンテキスト破綻 / durable execution / 保守コスト逆転）に到達したら ADR-0043 の再検討契機で再評価 |

### 9.7 計測と Phase 3 完了後の運用

3 者ループ全体の計測は §7.5（Evaluator 側）/ §8.7（Generator 側）の記録様式を踏襲し、Planner を含むループ単位で蓄積する。Planner のコストは小さい（V2 実測で約 0.4%、§4）ため、コスト主点は引き続き Generator（約 91%）。Phase 3 着手の Go/No-Go 前提は §8.6（最終 PASS 率 ≥ 80% 等・n ≥ 10）に従う。

到達点に達した以降は、記事の最重要教訓（§1）に従い、**モデル世代が更新されるたびに足場を再点検する**（効かなくなった足場を剥がし、新たに可能になった能力を足す）。スプリント機構（§8.3）・fail-close（§9.5）・ハーネス側モデル階層化（§8.5）・オーケストレーション手段（§9.4）はいずれもこの再点検の対象である。
