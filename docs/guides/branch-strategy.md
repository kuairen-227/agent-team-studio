# ブランチ戦略

GitHub Flow をベースにした軽量なブランチ戦略です。

## ブランチ命名規則

| プレフィックス | 用途 | コミット種別 |
| -------------- | ------ | ------------ |
| `feat/` | 新機能の開発 | `feat:` |
| `fix/` | バグ修正 | `fix:` |
| `docs/` | ドキュメント作業 | `docs:` |
| `chore/` | ツーリング・設定・メンテナンス | `chore:` |
| `refactor/` | リファクタリング | `refactor:` |
| `test/` | テスト追加・修正 | `test:` |

## ルール

1. **`main` ブランチは常にデプロイ可能な状態を維持する**
2. 作業は必ず `main` から分岐したブランチで行う
3. PR 経由でマージする（ソロ開発でも記録を残すため）
4. squash merge を推奨（履歴をクリーンに保つ）
5. マージ後はブランチを削除する

## フロー

```text
main ─────────────────────────────────── main
  └── feat/add-auth ── PR #3 ── merge ──┘
  └── fix/login-bug ── PR #4 ── merge ───┘
```

## 並行作業

複数ブランチで同時に作業する場合は Git Worktree を使用する。詳細は [worktree.md](./worktree.md) を参照。
