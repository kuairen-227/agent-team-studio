# docs/design/ 編集ルール

技術仕様・スキーマの SSoT。コードが存在する場合はコードが SSoT となり、ドキュメントはコード参照に留める。

## 置くもの

- 技術スタック・ライブラリ選定
- アーキテクチャ図・レイヤー構成・ディレクトリ設計
- API 設計規約（REST / WebSocket / 型共有）
- LLM 呼び出し方針（モデル選定・パラメータ・ストリーミング）
- データスキーマ（JSON Schema・Zod 定義・DB テーブル設計）
- I/O 契約（入力パラメータ型・内部 JSON 出力型・enum 値の定義）

## 置かないもの（product/ へ）

- ユーザーストーリー・受入基準
- 画面フロー・UI ラベル
- エージェントのペルソナ・プロンプト本文
- 失敗時の UX フロー
- ビジネス上の根拠・優先度判断

## コンプライアンスチェック

書いた後に以下が混入していないか確認する:

1. `画面` / `ボタン` / `非活性` / `エラー表示` が文脈なしで登場していないか
2. システムプロンプトが `` ```text `` ブロックで定義されていないか
3. 同一スキーマが `docs/product/` にも定義されていないか（重複禁止）

## 参照ルール（ADR-0021）

- テンプレート I/O スキーマは本ディレクトリが正式定義。product/ 側からの片方向参照のみ許容（双方向リンク禁止）
- 横方向参照（他 design/ doc）は SSoT 接続パターンのみ許容
- 同一 doc 内で同じリンク先を複数回参照しない（初出 1 回のみ）
