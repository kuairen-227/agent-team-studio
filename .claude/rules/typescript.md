# TypeScript コーディング規約

## JSDoc / TSDoc コメント方針

**書く場面:**
- ファイル冒頭のモジュール説明（目的・SSoT 参照）
- 公開 API の非自明な制約・副作用・エラー挙動
- 名前と型だけでは伝わらない設計上の理由

**書かない場面:**
- TypeScript の型で自明な情報の繰り返し（`@param name - string` 等）
- 関数名・型名を言い換えただけのコメント

**`@param` / `@returns` の使いどころ:**
挙動・制約・エラーの起点が名前と型だけでは読み取れない時のみ使う。型が十分に語っている場合は省略する。

**既存スタイルの踏襲:**
- 日本語で記述する
- 設計上の根拠・SSoT 参照（`docs/design/*.md`）を残す
- 「何をするか」より「なぜそうなのか」を書く
