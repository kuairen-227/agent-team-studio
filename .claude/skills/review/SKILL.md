# review

成果物をレビューする。`$ARGUMENTS` からレビューの種別と対象を判定し、適切な専門家エージェントに委譲する。

## 手順

### 1. レビュー種別の判定

`$ARGUMENTS` から種別と対象を判定する。

**明示的な指定がある場合**:

| 引数パターン | 種別 |
| ------------ | ------ |
| `product ...` / `requirements ...` | プロダクトレビュー |
| `design ...` | 設計レビュー |
| `code ...` | コードレビュー |
| `test ...` | テストレビュー |

**対象から推測する場合**:

| 対象 | 推測される種別 |
| ------ | ------------- |
| Issue 番号（`#XX`）、要件ドキュメント | プロダクトレビュー |
| `docs/adr/*`, `docs/design/*` | 設計レビュー |
| PR 番号、ソースコードファイル | コードレビュー |
| `*.test.*` ファイル | テストレビュー |

判別できない場合はユーザーに確認する。

### 2. 対象の読み取り

種別に応じてレビュー対象を取得する:

- **Issue**: `gh issue view <number>`
- **PR**: `gh pr view <number>` + `gh pr diff <number>`
- **ファイル**: 対象ファイルを Read で読み取る
- **ブランチの差分**: `git diff main...HEAD`

### 3. エージェントへの委譲

種別に応じて適切なエージェントに委譲する。複数エージェントが必要な場合は Agent ツールで並列に起動する。

| 種別 | エージェント | 各エージェントの役割 |
| ------ | ----------- | ----------------- |
| プロダクトレビュー | `po` + `pm` | po: ユーザー価値・JTBD整合・受入条件・スコープ / pm: 実現可能性・依存関係・リスク |
| 設計レビュー | `architect` | ADR間整合性・構造の妥当性・トレードオフ |
| コードレビュー | `architect` + `qa` | architect: コード構造・型安全性・命名・規約 / qa: エラー処理・セキュリティ・テスト充足度 |
| テストレビュー | `qa` | テスト設計・カバレッジ・境界値・テスト品質 |

エージェントへの委譲時に以下を渡す:

- レビュー対象の内容（ステップ 2 で取得した情報）
- レビューの文脈（関連する Issue、PR の目的、変更の背景）
- **レビューであること**を明示する（エージェントの評価基準のうちレビューに該当する観点を適用するよう指示）
- **レビュー粒度の指示**（下記参照）

#### レビュー粒度ガイドライン

種別ごとにレビューの焦点と除外事項を明示し、エージェントに渡す。工程に対して過剰に細かい指摘はタスクの進行を妨げるため、各種別の粒度を守る。

| 種別 | 焦点（指摘する） | 除外（指摘しない） |
| --- | --- | --- |
| プロダクトレビュー | ユーザー価値・JTBD 整合・受入条件の妥当性・スコープの過不足 | 技術的な実現方法・API 設計・データ構造 |
| 設計レビュー | 方針・判断の妥当性・ADR 間の整合性・トレードオフの明示・欠落した観点 | 実装の詳細（関数シグネチャ・エラーハンドリングの具体的手順・コード構造） |
| コードレビュー | 実装の正確性・型安全性・規約準拠・エラー処理・セキュリティ | ビジネス要件の妥当性（設計済みの前提） |
| テストレビュー | テスト設計・カバレッジ・境界値・テストの保守性 | プロダクションコードの設計判断 |

エージェントへの委譲プロンプトに「焦点」と「除外」を含めること。例: 設計レビューの場合、「方針・判断・ADR 整合性を評価してください。実装の詳細（関数の具体的な引数・エラー処理の手順等）には踏み込まないでください。」

### 4. レビュー結果の整形

エージェントからの結果を [template.md](template.md) の形式に整形して報告する。複数エージェントの場合は観点別に統合する。

### 5. レビュー結果の投稿

レビュー対象が PR または Issue の場合、レビュー結果をコメントとして投稿する。

- **PR の場合**: `gh pr comment <number> --body "<レビュー結果>"`
- **Issue の場合**: `gh issue comment <number> --body "<レビュー結果>"`
- **ファイルのみの場合**: コメント投稿は行わず、ユーザーへの報告のみ

## 注意点

- **判断はユーザーに委ねる**: レビュー結果を報告し、修正の判断はユーザーが行う
- **スコープを守る**: レビュー対象外のコードに対する指摘を混ぜない
- **良い点を必ず含める**: 指摘だけでなく、良い判断・良い実装も報告する
- **複数エージェントの重複排除**: 同じ指摘が複数エージェントから出た場合は統合する
