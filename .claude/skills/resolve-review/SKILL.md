# resolve-review

`$ARGUMENTS` で指定された PR のレビュー指摘を読み取り、各指摘を評価・判定したうえで修正を実装する。

## 手順

### 1. PR の確認とブランチチェック

```bash
gh pr view <pr-number> --json number,title,headRefName,baseRefName,state,url
```

- PR が open であることを確認する。closed/merged の場合は報告して終了
- `git branch --show-current` で現在のブランチと `headRefName` を比較する。異なる場合はユーザーに通知し `git switch <headRefName>` を提案する。自動切り替えはしない
- `git status` で未コミット変更を確認する。ある場合は警告する

### 2. レビューコメントの取得

`gh pr view` でコメントとレビューを取得する:

```bash
# PR のコメント・レビュー・インラインコメントを取得
gh pr view <pr-number> --json comments,reviews,reviewThreads

# 読みやすい形式でコメントを確認
gh pr view <pr-number> --comments
```

コメントが存在しない場合は報告して終了。

### 3. コメントの解析と分類

取得したコメントを個別の指摘項目に分解する。

**テンプレート形式の検出**:

review スキルの出力形式（`🔴 Must` / `🟡 Should` / `⚪ Nit` テーブル）を検出した場合、テーブルの各行を個別の指摘として解析し、重要度を引き継ぐ。

**インラインコメント**:

ファイルパス・行番号メタデータ付きで1件1指摘として扱う。

**自由形式コメント**:

1コメント1指摘として扱う。言語から重要度を推測する（明確でなければ `should` とする）。

**除外対象**:

以下はレビュー指摘として扱わずスキップする:

- LGTM、承認コメント
- ボットのステータス通知（CI 結果等）
- 質問のみで改善要求を含まないコメント
- resolved 済みのレビュースレッド

### 4. 各指摘のコード文脈確認と判定

各指摘について:

1. 関連ファイル・コード領域を特定する（インラインコメントのメタデータ、またはコメント本文中のファイル・行参照から）
2. 該当コードを Read で読み取り、現在の状態を把握する
3. `gh pr diff <pr-number>` で変更文脈を確認する
4. 以下の判定を付与する:

| 判定 | 意味 | 対応案 |
| --- | --- | --- |
| `要対応` | 指摘が妥当、コード変更が必要 | 具体的な修正方針を記載 |
| `対応済み` | 後続コミット等で既に修正されている | — |
| `非該当` | 現在のコードに当てはまらない（誤読・文脈違い等） | 理由を記載 |
| `要議論` | トレードオフがあり判断が必要 | 選択肢を提示 |

**実装方針の優先順位**:

修正方針を立てる際、以下の順で検討する:

1. **削れないか**: 指摘箇所そのもの（または周辺）を削る／統合する解で済まないか
2. **命名・構造で語れないか**: コメント追加・ログ追加・変数追加で説明する前に、リネーム・分割・型表現で意図が伝わらないか
3. **追加で解決**: 上記で済まない場合のみ、新しいコード・コメント・ガードを追加する

特に「コメントを追加せよ」「説明的なログを足せ」「抽象化レイヤーを設けよ」系の指摘は、まず「そもそも追加せず命名・構造で解決できるか」を検討し、できる場合は対応案として提示する（指摘そのものは尊重しつつ、より本質的な代案を出す）。

競合する指摘（2つの指摘が矛盾する変更を求める場合）はフラグし、どちらに従うかユーザーに選択を求める。

### 5. ユーザーへの評価結果の提示

全指摘の評価結果を一覧で提示する:

```markdown
## レビュー指摘の評価 — PR #XX

**対象コメント**: X件（must: X / should: X / nit: X）

| # | 重要度 | 指摘内容（要約） | 判定 | 対応案 |
|---|--------|-----------------|------|--------|
| 1 | 🔴 must | [要約] | 要対応 | [対応案] |
| 2 | 🟡 should | [要約] | 対応済み | — |
| 3 | ⚪ nit | [要約] | 非該当 | [理由] |
```

`要議論` の項目がある場合は別途選択肢を提示する。

ユーザーの承認を待つ。ユーザーは以下が可能:

- 対応項目の選択・除外
- 判定の上書き（例: `非該当` → `要対応`）
- `要議論` 項目の方針決定

### 6. 承認された修正の実装

ユーザーが承認した `要対応` 項目について:

1. 該当コードを修正する
2. 関連する修正は1コミットにまとめ、独立した修正は分離する
3. `bun run lint` / `bun run type-check` で検証する（プロダクトコードの変更がある場合）
4. CLAUDE.md のコミット規約に従いコミットする
   - プレフィックス: `fix:` または `refactor:`（修正内容に応じて選択）
   - スコープは変更対象に応じて付与する（例: `fix(project):`, `refactor(api):`）
   - コミットメッセージに `#<pr-number>` を含める

### 7. 完了報告と結果投稿

**ユーザーへの報告**:

- 実施した変更の概要
- スキップした項目とその理由
- 次のステップの提案（push、re-request review 等）

**PR へのコメント投稿**:

対応結果サマリを PR コメントとして投稿する:

```bash
gh pr comment <pr-number> --body "<対応結果>"
```

投稿フォーマット:

```markdown
## レビュー対応完了

| # | 指摘 | 対応 |
|---|------|------|
| 1 | [指摘要約] | ✅ 修正済み（commit SHA） |
| 2 | [指摘要約] | ⏭️ 対応済み |
| 3 | [指摘要約] | ❌ 非該当（理由） |
| 4 | [指摘要約] | 💬 議論の結果スキップ |
```

## 注意点

- **判断はユーザーに委ねる**: 評価結果を報告し、修正の最終判断はユーザーが行う
- **スコープを守る**: レビュー指摘の範囲外の改善を混ぜない
- **競合する指摘**: フラグして選択を求める。勝手に片方を選ばない
- **追加より削減を優先**: 指摘対応で新コード・新コメントを足すより、削る・命名で語る解を優先検討する（CLAUDE.md 文体規約と整合）
